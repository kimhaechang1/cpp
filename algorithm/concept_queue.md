# 0x06: 큐 (Queue) 와 메모리 캐시 (Cache)

## 🎮 실무 프로젝트 상황: 네트워크 통신과 패킷 처리
FPS 게임 서버에는 1초에도 수백, 수천 개의 플레이어 이동 패킷이 도착합니다. 
이 패킷들을 **"도착한 순서대로(First In, First Out = FIFO)"** 처리해야만 플레이어의 위치가 텔레포트하지 않고 자연스럽게 이동합니다. 이렇게 "먼저 들어온 데이터가 먼저 나가는" 규칙을 가진 자료구조가 바로 **큐(Queue)** 입니다.

---

## 💡 이론: 큐(Queue)의 핵심 원리

큐는 파이프(Pipe) 또는 양쪽이 뚫린 터널을 생각하면 됩니다.
*   **Push (삽입)**: 터널의 뒷부분(Rear/Tail)으로 데이터가 들어갑니다.
*   **Pop (삭제)**: 터널의 앞부분(Front/Head)에서 데이터가 빠져나옵니다.

### 🧠 Data-Oriented Design (데이터 지향 설계) 관점에서의 큐
연결 리스트(Linked List) 단원에서 배운 야매 연결 리스트를 기억하시나요? 
큐 역시 메모리 파편화를 유발하는 동적 할당 노드(`Node*`) 대신, **거대한 배열 1개**와 위치를 가리키는 **인덱스 2개(head, tail)**만으로 구현하는 것이 현대 게임 엔진의 정석(Data-Oriented Design)입니다.

이렇게 배열(Array) 덩어리로 만들면, CPU가 캐시(Cache Memory)를 읽어 들일 때 주변의 다음 패킷 데이터까지 한꺼번에 캐시로 긁어오기(Prefetching) 때문에 **캐시 적중률(Cache Hit)이 극단적으로 높아져** 초고속 패킷 처리가 가능해집니다.

### 💻 배열을 이용한 큐 구현 원리 (선형 큐)

```cpp
const int MX = 1000005;
int dat[MX];
int head = 0;  // 빠져나갈 데이터의 위치
int tail = 0;  // 다음 데이터가 들어갈 위치

// 데이터 추가 (Push)
void push(int x) {
    dat[tail] = x;
    tail++;
    // 축약: dat[tail++] = x;
}

// 데이터 추출 (Pop)
void pop() {
    head++; // 배열의 데이터를 진짜로 지우지 않습니다! 그냥 시작점 위치만 뒤로 미룹니다.
}
```

#### 🚨 선형 큐의 치명적 약점과 '원형 큐(Circular Queue)'
위 방식대로 큐를 쓰다 보면, `head`와 `tail`이 계속 오른쪽으로만 이동합니다. 결국 거대한 배열의 끝(`MX`)에 도달하면 더 이상 쓸 수 없는 버려진 큐가 됩니다. 

이 문제를 해결하기 위해 **원형 큐(Circular Queue)**가 등장했습니다. 배열의 끝에 도달하면 다시 0번 인덱스(`% MX` 연산 활용)로 돌아오게 만드는 방식인데, 코딩 테스트(알고리즘)에서는 범위(명령의 수)가 주어지기 때문에 대개 크기를 넉넉히 잡고 **단순 선형 큐** 구조로 풀어도 전혀 문제가 없습니다. (실무 엔진 개발에서는 원형 큐나 링 버퍼 Ring Buffer 구조를 사용합니다.)

---

## ❓ 큐가 STL과 만났을 때
C++ 표준 라이브러리에는 `<queue>`가 존재하며, 내부적으로는 덱(`std::deque`) 구조를 사용해 양방향 접근이 최적화되어 있습니다. 
하지만 알고리즘 기초 과정의 **[필수 학습 규칙]**에 따라, 첫 문제는 무조건 `STL`을 쓰지 않고 우리가 직접 배열과 `head`, `tail` 인덱스를 조작하여 풀어보면서 원리를 체득할 것입니다!
