# [Module 08] STL(표준 템플릿 라이브러리) 컨테이너: 게임 개발의 핵심 무기고

## 1. 실무 상황: 바퀴를 다시 발명하지 마라 (Don't reinvent the wheel)
앞서 배운 템플릿 기술을 이용하면 리스트(List)나 가변 배열(Vector)을 우리가 직접 바닥부터 짜서 쓸 수도 있습니다. 실제로 아주 옛날에는 회사마다 자체 리스트 클래스를 만들어 썼습니다.

하지만 수십 년간 천재 수학자들과 C++ 엔지니어들이 영혼을 갈아 넣어 "C++에서 이것보다 더 빠르게 작동하는 자료구조는 수학적으로 불가능하다" 수준으로 만들어 놓은 템플릿 묶음이 존재합니다. 그것이 바로 **STL(Standard Template Library)**입니다.

우리는 게임을 만들 때 뻔한 자료구조 코딩에 시간을 낭비할 필요 없이, 이 무기고(STL)에서 필요한 무기(컨테이너)를 꺼내서 전략적으로 조립하기만 하면 됩니다.

---

## 2. 최고 존엄 무기: `std::vector` (동적 배열)
C++ 게임 개발 실무에서 데이터를 담아야 할 때 **95% 이상의 상황에서 무조건 1순위로 채택**하는 가장 중요한 녀석입니다.

**[특징]**
- 연속된 메모리 덩어리(Block) 구조입니다. 
- 이전에 Module 7에서 `AoS vs SoA` 파트에서 배웠듯, **CPU 캐시 히트(Cache Hit)** 율이 현존하는 모든 자료구조 중 가장 높아서 순회 처리 속도가 압도적으로 빠릅니다. (Data-Oriented Design의 핵심)
- 필요에 따라 크기(Capacity)가 가변적으로 늘어납니다.

```cpp
#include <vector>

std::vector<int> numbers; // int 전용 가변 배열 (템플릿이므로 T 위치에 int 주입)
numbers.push_back(10);    // 맨 뒤에 데이터 추가
```

## 3. 영혼의 단짝: `std::pair`
C++ STL 컨테이너들을 다루다 보면 필연적으로 두 개의 데이터를 한 쌍(Pair)으로 묶어서 던져줘야 할 일이 밥 먹듯이 생깁니다. 
좌표 $(X, Y)$ 일 수도 있고, 아이템의 $(ID, 이름)$ 구조일 수도 있습니다. 이때 `struct` 구조체를 귀찮게 새로 만들지 않고 임시로 두 개의 템플릿(`T1`, `T2`)을 묶어주는 녀석이 `std::pair` 입니다.

```cpp
#include <utility>
#include <iostream>

// T1=int, T2=std::string 인 1회용 묶음
std::pair<int, std::string> myItem(1001, "Red Potion");

std::cout << "ID: " << myItem.first << std::endl;   // 첫 번째 인자 접근
std::cout << "Name: " << myItem.second << std::endl; // 두 번째 인자 접근
```

## 4. `std::map` vs `std::unordered_map` (사전/딕셔너리 구조)
게임을 만들 때 "1001번 고블린 내놔", "RedPotion 아이템 객체 줘" 처럼 데이터 뭉텅이 안에서 **특정 키워드(Key)로 값(Value)을 빠르게 검색**해야 하는 경우가 매우 많습니다. 이때는 두 가지 선택지가 있습니다. 

질문해 주신 내용에 완벽하게 부합하는 핵심 개념입니다.

### (1) `std::map` (정렬 됨 - Ordered)
*   **작동 원리**: 내부적으로 이진 탐색 트리(Red-Black Tree)라는 복잡한 구조로 데이터를 저장합니다.
*   **특징**: 데이터를 넣을 때 **무조건 오름차순으로 자동 정렬**됩니다.
*   **검색 속도**: $O(\log N)$ (수십만 건의 데이터라도 나무 가지를 반갈죽 치며 빠르게 찾음)
*   **쓰는 상황**: "유저의 랭킹(랭킹 점수순 정렬)"을 유지하면서 보여줘야 하거나, 아이템을 "가나다 순"으로 정렬된 상태로 유지해야 할 때 씁니다.

### (2) `std::unordered_map` (정렬 안 됨 - Unordered)
*   **작동 원리**: 내부적으로 **해시 테이블(Hash Table)** 이란 걸 사용하여 메모리 공간의 번지수에 바로 직행합니다.
*   **특징**: 아무리 데이터를 넣어도 "정렬되지 않습니다". 순서가 완전 뒤죽박죽 쓰레기처럼 담깁니다.
*   **검색 속도**: $O(1)$ (데이터가 100만 건이든 1000만 건이든 **무조건 단 번에(상수 시간)** 찾아냅니다. 현존하는 최고 속도)
*   **쓰는 상황**: 아이템 ID 도감(Dictionary)이나 몬스터 DB처럼 **정렬이고 나발이고 그딴 건 필요 없고, 오직 "존재하는지만 0.0001초 안에 검색해서 찾아줘"** 라는 게임 속도 최적화 상황에서 주로 씁니다.

```cpp
#include <map>           // std::map
#include <unordered_map> // std::unordered_map
#include <string>
#include <iostream>

int main() {
    // 1. 순서가 보장되는 Map
    std::map<int, std::string> orderedMap;
    orderedMap.insert(std::make_pair(3, "C")); // 데이터 넣을때 std::pair를 활용
    orderedMap[1] = "A"; // 이렇게 넣어도 됨
    orderedMap[2] = "B";

    // 출력해보면 넣은 순서(3->1->2)가 아니라 무조건 (1->2->3) 으로 자동 정렬되어 나옴

    // 2. 순서를 무시하는 초고속 Map
    std::unordered_map<int, std::string> fastMap;
    fastMap[8] = "Orc";
    fastMap[1] = "Slime";
    fastMap[300] = "Dragon";

    // "Dragon" 이 있는지 찾기위해 단계를 안 거치고 [300] 이라는 숫자에서 즉시 산술계산(해시)하여 튀어나옴
    std::cout << fastMap[300] << std::endl; 

    return 0;
}
```

## FAQ: 또 다른 컨테이너 & 클래스인데 소문자인 이유
**Q1: 알고리즘(Array, Stack, Queue, Linked List 등)에서 배운 것 말고 게임 개발을 위해 추가로 알아야 할 STL이 더 있나요?**

**A1: 거의 다 배우셨습니다! 추가로 딱 2개만 눈여겨보시면 됩니다.**
1. **`std::set` / `std::unordered_set`**: 아까 배운 `map`에서 'Value'를 빼고 오직 **'Key(중복 없는 고유한 키값)'만 저장**하는 자료구조입니다. (예: "이미 방문한 타일인지 검사할 때", "중복 없는 고유한 플레이어 ID 목록을 만들 때" 씁니다.)
2. **`std::priority_queue` (우선순위 큐)**: 가장 가치가 높거나 낮은 것(최댓값/최솟값)이 항상 맨 앞으로 튀어나오는 마법의 큐입니다. 게임의 알고리즘 길찾기(**A* Pathfinding**)에서 압도적인 효율을 내어 몬스터 AI 방황 로직 구현 시 밥 먹듯이 쓰입니다.

**Q2: `vector`, `map` 이런 애들은 분명히 `class` 인데 왜 C++ 코딩 컨벤션인 대문자 파스칼 케이스(PascalCase, 예: `Map`, `Vector`)로 시작하지 않고 싹 다 소문자로 만들어져 있나요?**

**A2: C++의 역사적 설계 철학(C언어와의 통일성) 때문입니다!**
C++ 표준 라이브러리(STL)의 창시자들과 C++ 위원회는 `std::vector` 나 `std::string` 같은 템플릿 클래스들이, 평범한 '외부 라이브러리'나 개인이 만든 '일반 클래스'처럼 보이기를 극도로 원치 않았습니다.

대신, 이 녀석들이 마치 태초부터 C언어에 존재했던 **기본 내장 자료형(`int`, `float`, `double`, `char`) 들과 완벽하게 동일한 격(Level)**으로 취급받고 자연스럽게 소스 코드에 섞여 들어가기를 원했습니다.
그래서 `int` 나 `float` 이 소문자인 것처럼, C++의 가장 근간이 되는 기반 언어급 표준 객체들도 모두 소문자(snake_case)로 명명하자는 룰이 정해진 것입니다. 

(그래서 현대의 프로그래머들은 **'C++ 표준과 관련된 건 경외심을 담아 소문자로 두고, 내가 직접 만든 게임의 비즈니스 클래스는 대문자 섞어 쓰기(PascalCase, `Monster`, `Inventory`)로 철저히 구분 짓자'** 고 약속하여 가독성을 높입니다!)
