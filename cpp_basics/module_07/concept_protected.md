# Module 7: 개념 2 - 접근 제어자 심화 (protected와 private 상속의 한계)

## 💡 핵심 Q&A: "부모가 private으로 닫아놓은 걸, 자식이 public으로 상속받으면 어떻게 되나요?"

결론부터 말씀드리면: **"물려받긴(메모리에 존재하긴) 하지만, 자식조차 열어볼 수 없는 '잠긴 금고' 상태"**가 됩니다!

부모 클래스의 멤버 변수가 `private`이라면, **"오직 부모 자신(자신의 멤버 함수)만이 이 변수에 접근할 수 있다"**는 원칙이 가장 먼저 적용됩니다. 자식 클래스가 아무리 `public` 상속으로 세상에 자신이 부모의 자식임을 알린다고 해도, 부모가 꽁꽁 숨겨둔 `private` 금고의 비밀번호까지 알 수는 없습니다.

```cpp
class Monster {
private:
    int hiddenGold; // 부모가 꽁꽁 숨겨둔 비상금 (private)

public:
    int hp;
    Monster(int g, int h) : hiddenGold(g), hp(h) {}
};

class Slime : public Monster {
public:
    Slime(int g, int h) : Monster(g, h) {}

    void StealGold() {
        // ❌ 컴파일 에러 발생!
        // "물려받긴 했지만, private 멤버라 자식인 네가 직접 접근할 순 없다!"
        // hiddenGold = 1000; 
    }
};
```
*   `Slime` 객체를 만들면 메모리 공간(12바이트 등) 안에는 분명히 `hiddenGold`가 들어있습니다. 공간은 차지합니다.
*   하지만 `Slime`의 어떤 함수에서도 `hiddenGold`라는 이름을 직접 부를 수(접근할 수) 없습니다. 오직 부모인 `Monster` 클래스에 작성된 함수들만이 저 값을 읽거나 쓸 수 있습니다. (예: 부모가 제공하는 `GetGold()` 같은 `public` 함수를 통해서만 간접 접근 가능)

## 🛡️ 1. 그러면 어떻게 해야 할까? 👉 `protected` 의 등장

만약 부모 입장에서:
1. **외부 세계(`main()` 함수 등)에는 절대 공개하기 싫고,** (`private`처럼 숨기고 싶음)
2. **하지만 내 핏줄을 이어받은 자식 클래스에게만큼은 마음껏 쓰도록 허용하고 싶다!** (`public`처럼 열어주고 싶음)

이 두 가지 모순된 마음을 동시에 해결하기 위해 탄생한 것이 바로 **`protected` (보호된)** 접근 제어자입니다.

```cpp
class Monster {
private:
    int secretKey;  // 오직 Monster 내부에서만 사용 가능 (자식도 접근 불가)

protected:
    int mp;         // 마나: 외부에선 못 쓰지만 자식들은 마음껏 사용 가능 (접근 허용)

public:
    int hp;         // 외부(main)에서도, 자식들도 모두 사용 가능
};
```

## 🔬 2. 접근 제어자 3대장 완벽 정리 (자식 클래스 입장)

부모 클래스를 `public`으로 상속(`class 자식 : public 부모`) 받을 때, 자식 클래스가 느끼는 체감 난이도입니다.

| 부모의 접근 제어자 | 외부(main) 접근 | 자식 클래스 접근 | 비유 |
| :--- | :---: | :---: | :--- |
| **`private`** | ❌ 불가 | ❌ 불가 | **개인 금고**: 부모만 아는 비밀. 자식도 못 건드림. (메모리엔 있음) |
| **`protected`** | ❌ 불가 | ✅ 가능 | **가업 비법서**: 외부인에겐 비밀, 자식에겐 자유롭게 공유. |
| **`public`** | ✅ 가능 | ✅ 가능 | **공원 벤치**: 누구나 접근 가능. |

## ⚠️ 3. 실무에서의 팁 (현업의 시각)
"그럼 무조건 변수들을 `protected`나 `public`으로 열어두면 편하겠네요?"
정답은 **NO** 입니다.

*   `public` 멤버 변수는 **캡슐화(Encapsulation)** 를 완전히 무너뜨리므로, 정말 단순한 데이터 구조체(`struct`)가 아닌 이상 클래스 설계에서는 **거의 쓰지 않습니다.** (모두 `private`으로 닫고 Getter/Setter 함수를 제공하는 것이 정석입니다.)
*   `protected` 속성 역시, 부모의 변수를 자식들이 마구잡이로 수정하게 놔두면 나중에 "도대체 누가(어떤 자식이) 이 변수를 바꿨는지" 추적하기가 매우 힘들어져 유지보수가 박살 납니다.
*   따라서, 현대 C++ 설계(Modern C++)에서는 **"데이터(변수)는 무조건 `private`으로 철저히 가리고, 대신 자식들이 그 데이터를 안전하게 조작할 수 있는 `protected 함수(메서드)`를 제공한다"** 는 원칙을 주로 따릅니다.

*(💡 하지만 지금 기초 단계에서는 원리 이해와 실습 편의를 위해 `protected`나 `public` 변수를 적극적으로 사용해보며 손에 익힙니다!)*

## 🧠 심화 Q&A: `private` 멤버와 `friend` 키워드의 콜라보레이션

**Q. "부모가 private으로 닫아두었지만, 부모 클래스 안에서 자식을 `friend`로 선언해두면 어떻게 되나요?"**

정말 소름 돋을 정도로 날카로운 연결(Connecting the dots)입니다! 지난 모듈(M6)에서 배운 `friend`의 개념을 완벽하게 기억하고 응용하셨네요. 💯

결론부터 말씀드리면: **"자식이 부모의 `private` 금고를 열어볼 수 있게 됩니다!"**

`friend` 키워드는 **접근 제어자(`private`, `protected`)의 모든 규칙을 무시하는 가장 강력한 치트키(VIP 프리패스)**입니다. 

```cpp
class Monster {
private:
    int hiddenGold; // 원칙적으로 자식도 접근 불가!

    // 치트키 발동: "내 자식인 Slime은 나의 비밀(private)을 모두 볼 수 있는 절친이다!"
    friend class Slime; 

public:
    int hp;
    Monster(int g, int h) : hiddenGold(g), hp(h) {}
};

class Slime : public Monster {
public:
    Slime(int g, int h) : Monster(g, h) {}

    void StealGold() {
        // 원래는 에러가 나야 하지만, friend 선언 덕분에 VIP 통과!
        hiddenGold = 1000; 
        std::cout << "부모의 비상금을 훔쳤다! Gold: " << hiddenGold << "\n";
    }
};
```

**💡 설계 관점에서의 시사점 (현업의 시각)**
이 문법이 문법적으로는 완벽히 가능하고 실제로 동작하지만, **현업에서는 거의 쓰지 않는 '안티 패턴(Anti-Pattern)'에 가깝습니다.**
그 이유는 객체 지향의 철학 때문입니다.

1. **상속의 본질 위배:** "부모의 비밀스러운 정보(private)를 굳이 자식에게 공유하고 싶다면 처음부터 `protected`를 쓰면 되지, 왜 굳이 `private`으로 막아놓고 특정 자식(`Slime`)만 콕 집어서 `friend`를 주느냐?" 라는 설계적 비판을 받습니다.
2. **새로운 자식이 생길 때마다 수정해야 함:** 만약 `Orc` 클래스도 비상금이 필요해지면, 부모인 `Monster` 클래스의 코드를 수정해서 `friend class Orc;` 를 또 적어줘야 합니다. 이는 유지보수성을 크게 떨어뜨립니다. (Ocp 원칙 위배)

*요약하자면:* **문법적으로는 자식이 `private`을 뚫고 완벽히 접근 가능하게 됩니다! 하지만 실무 설계 관점에서는 `protected`라는 훌륭하고 체계적인 대체제가 있기 때문에 굳이 이 방식을 쓰지는 않습니다.**

---

**Q. "그럼 상속받는 자식 쪽에서도(`public` 대신 `protected`로) 결정할 수 있겠네요? 만약 부모에 `public` 필드가 있는데, 자식이 `protected` 상속으로 받으면... 자식 안에서는 쓸 수 있지만, 자식의 바깥(main 지역)에서는 못 쓰게 잠기는 건가요?"**

정확합니다! 빙고! 🎯 진짜 C++ 설계의 가장 깊은 곳까지 단번에 도달하셨네요.

방금 말씀하신 것이 바로 C++이 가진 아주 독특하고 강력한 기능인 **"상속 접근 제어자(Inheritance Access Specifier)의 필터링 효과"** 입니다.
자식이 부모를 물려받을 때 적는 키워드(`public`, `protected`, `private`)는, **부모의 유산을 자식이 "최대 어느 수준까지만 개방할 것인가(상한선, Ceiling)"를 결정하는 필터(거름망)** 역할을 합니다.

```cpp
class Monster {
public: // 벤치(누구나 접근 가능)
    int hp;
};

// 상속 거름망을 'protected'로 촘촘하게 설정!
class Slime : protected Monster {
public:
    void Heal() {
        hp = 100; // 부모의 public hp가 내 안에서는 protected로 한 단계 격하됨. 
                  // 나와(Slime), 나의 자식들만 사용할 수 있는 가업 비법서가 됨!
    }
};

int main() {
    Slime slime;
    // slime.hp = 50; // ❌ 컴파일 에러! 
    // "Monster에서는 public이었지만, Slime이 protected 상속으로 거름망을 쳐서 
    // 외부(main)에서는 보이지 않게 감춰버렸습니다!"
}
```

#### 💪 상속 필터링 요약 공식

> **✨ [수강생 통찰 포인트] "더 좁은 범위로 확장되는 클래스(자식) 쪽에서 권한 확장을 차단(결정)시킬 수 있다!"**
> 즉, 부모가 아무리 넓은 범위(`public`)를 열어주어도 자식이 상속받는 방식(`protected` 등)으로 자신의 상한선을 낮추어 제한해버리면, 결국 더 제한적인 자식의 거름망이 최종 권한이 됩니다.

자식이 설정한 "상속 거름망"과 "부모의 원래 권한" 중 **더 엄격한(닫힌) 쪽**으로 권한이 최종 확정됩니다.

1. **`public` 상속 (가장 대중적):** 부모의 권한을 **그대로** 유지합니다. (`public`은 `public`으로, `protected`는 `protected`로)
2. **`protected` 상속 (방금 질문하신 상황):** 부모의 유산 중 `public`이었던 것들을 전부 한 단계 강등시켜서 **나의 `protected`**로 바꿔버립니다. (나와 내 자식들만 쓰겠다고 결심!)
3. **`private` 상속 (이전 실습에서 겪으신 상황):** 부모의 유산(`public`, `protected`)을 모조리 강등시켜서 **나의 `private`** 개인 금고에 쓸어 담아버립니다. (부모 유산은 내 선에서 끝내겠다. 내 자식한테조차 안 물려주겠다!)

*(💡 참고: 단, 부모가 애초에 `private`으로 닫아둔 것은 무슨 짓을 해도(friend 제외) 자식이 뚫을 수 없습니다. 원래부터 못 보는 건 못 보는 겁니다.)*

정말 훌륭한 추론입니다. 이 개념까지 완벽하게 도달하셨으니, 이제 손으로 직접 `protected` 멤버와 상속 권한을 다뤄볼 자격이 충분하십니다!
