# [Module 09] 이동 시맨틱 (Move Semantics): 자원을 '복사'하지 말고 '강탈'하라

## 1. 왜 이동(Move)이 필요한가?

지금까지 우리는 데이터를 넘길 때 **복사(Copy)**를 주로 썼습니다. 하지만 엄청나게 큰 데이터를 가진 객체라면 어떨까요?

```cpp
std::vector<int> bigData(100'000'000, 7); // 400MB 정도의 데이터
std::vector<int> otherData = bigData;     // 400MB를 새로 할당하고 복사! CPU/메모리 비명!
```

만약 `bigData`를 더 이상 쓰지 않을 예정이라면, 굳이 새로 복사할 필요가 있을까요? 그냥 `bigData`가 가진 메모리 주소(포인터)만 쏙 빼앗아 오면 되지 않을까요?

이것이 바로 **이동 시맨틱(Move Semantics)**의 핵심입니다.

---

## 2. L-value vs R-value

이동을 이해하려면 먼저 두 가지 타입을 구분해야 합니다.

| 타입 | 의미 | 특징 |
|------|------|------|
| **L-value** | 이름을 가진 실체 (위치 있음) | 주소(`&`)를 취할 수 있음. 대입 연산자의 왼쪽 가능. |
| **R-value** | 이름 없는 임시 값 (우측 값) | 사용 후 즉시 소멸됨. 주소를 취할 수 없음. |

```cpp
int a = 10;      // a는 L-value, 10은 R-value
int b = a + 5;   // b는 L-value, (a+5)는 R-value (임시 계산 결과)
```

C++11부터는 이 **R-value**를 가리키는 특별한 참조자인 **R-value 참조(`&&`)**가 등장했습니다.

---

## 3. `std::move` — "나 이제 이거 안 쓸 거야, 가져가!"

`std::move`는 이름은 거창하지만, 실제로는 아주 단순한 일을 합니다.
> "L-value를 **R-value로 캐스팅**해준다."

즉, 컴파일러에게 "이 변수는 이제 곧 죽을 녀석(임시 객체)처럼 취급해도 돼!"라고 알려주는 신호입니다.

```cpp
std::string str1 = "Large String Data...";
std::string str2 = std::move(str1); // str1을 임시 객체처럼 취급해서 str2로 '이동'

// 결과:
// str2는 데이터를 안전하게 가로챔.
// str1은 '빈 상태' (유효하지만 정의되지 않은 상태)가 됨.
```

## 4. 이동 생성자 (Move Constructor)

클래스에서 이동 시맨틱을 지원하려면 **이동 생성자**가 필요합니다.

```cpp
class Buffer {
    int* data;
    size_t size;

public:
    // 1. 복사 생성자 (깊은 복사 - 느림)
    Buffer(const Buffer& other) : size(other.size) {
        data = new int[size];
        std::copy(other.data, other.data + size, data);
    }

    // 2. 이동 생성자 (자원 가로채기 - 매우 빠름!)
    Buffer(Buffer&& other) noexcept 
        : data(other.data), size(other.size) { // 주소만 쏙 빼옴
        other.data = nullptr; // 원본의 연결고리를 끊음 (중요! 안 하면 Double Free)
        other.size = 0;
    }

    ~Buffer() { delete[] data; }
};
```

### 💡 이동의 3단계 공식
1.  상대방(`other`)의 자원을 내 것으로 **가져온다** (포인터 복사).
2.  상대방의 자원을 **null** 등으로 초기화한다 (파괴자 방지).
3.  이 작업은 예외가 없음을 보장한다 (`noexcept`).

## 5. 이동 대입 연산자 (Move Assignment Operator)

이미 존재하는 객체에 다른 객체의 자원을 뺏어다 넣을 때 필요한 연산자입니다.

```cpp
Buffer& operator=(Buffer&& other) noexcept {
    // 1. 자가 대입 방지 (Self-assignment check)
    // - 나 자신을 나에게 이동시키는 뻘짓(?)을 막습니다. 
    if (this != &other) {
        // 2. 기존 자원 해제 (Cleanup)
        // - 대입은 '기존 것을 버리고 새 것을 받는' 과정입니다. 
        delete[] data; 

        // 3. 자원 가로채기 (Steal)
        data = other.data;
        size = other.size;

        // 4. 상대방 무력화 (Disarm)
        other.data = nullptr;
        other.size = 0;
    }
    return *this;
}
```

### ❓ 이동 생성자와의 결정적인 차이점
*   **이동 생성자**: 빈 땅에 집을 짓는 것. (기존 것이 없으므로 해제가 필요 없음)
*   **이동 대입 연산자**: 헌 집을 부수고 새 집을 짓는 것. (**`delete`** 과정이 반드시 필요!)

---

## 6. 요약 및 주의사항

1.  **속도 향상**: 복사(Heap 할당 + 반복문 복사)보다 이동(포인터 대입)이 압도적으로 빠릅니다.
2.  **unique_ptr**: 복사가 불가능한 `unique_ptr`은 `std::move`를 통해서만 소유권을 바꿀 수 있습니다.
3.  **이동 후 상태**: `std::move` 당한 변수는 이제 "껍데기"만 남은 상태이므로 다시 사용하지 않는 것이 원칙입니다.

## 6. Q&A: 심화 질문과 답변

### Q1. "결국 주소 뺏고 nullptr 넣는 게 다인데, 왜 굳이 &&를 쓰나요?"
매우 날카로운 질문입니다! 실제로 이동의 **내부 동작(Mechanism)**은 주소 복사와 원본 초기화가 전부입니다. 하지만 **&&**는 이를 실현하기 위한 **표준화된 통로** 역할을 합니다.

1.  **바인딩 규칙 (Binding Rule)**: 일반 참조(`&`)는 임시 객체(R-value)를 아예 받을 수 없도록 문법적으로 막혀 있습니다.
2.  **수정 권한 (Modifiability)**: 상수 참조(`const &`)는 임시 객체를 받을 수는 있지만, 그 안의 데이터를 `nullptr`로 수정(강탈)할 수 없습니다.
3.  **안전한 강탈**: 결국 `&&`는 **"임시 객체를 받을 수 있으면서, 동시에 내용을 수정할 수 있는 유일한 대안"**입니다.

### Q2. "이동하고 나면 원본(other)은 즉시 소멸되나요?"
아닙니다! 원본 객체는 자기가 속한 스코프(중괄호)가 끝날 때까지 스택에 버팁니다. 다만 그 안의 **'알맹이(자원)'**만 쏙 빠져나간 **'빈 껍데기'** 상태가 될 뿐입니다. 
우리가 `nullptr`을 넣어주는 이유는, 나중에 이 껍데기가 죽을 때(소멸자 호출 시) 우리가 이미 뺏어온 자원을 함께 파괴하지 못하도록 **무기를 해제(Disarm)**하는 작업입니다.

### Q3. "이동 생성자와 이동 대입 연산자는 언제 각각 호출되나요?"
*   **초기화 (Initialization)**: 새로운 변수가 탄생할 때! (`=`) 기호를 쓰더라도 실체는 **이동 생성자**입니다.
    `HugeBuffer b2 = std::move(b1);` ➞ `HugeBuffer b2(std::move(b1));`와 동일
*   **대입 (Assignment)**: 이미 존재하던 객체에 값을 다시 넣을 때 호출됩니다.
    `b2 = std::move(b3);` ➞ `b2.operator=(std::move(b3));` 호출

### Q4. "noexcept 키워드는 왜 그렇게 중요한가요?"
`std::vector` 같은 STL 컨테이너들은 **'강한 예외 보증(Strong Exception Guarantee)'**을 철저히 지킵니다.

1.  벡터의 크기가 늘어나서 자리를 옮길 때, 만약 **이동(Move)** 도중 예외가 터지면 원본 데이터가 유실될 위험이 있습니다.
2.  이런 위험을 방지하기 위해 벡터는 이동 생성자에 **`noexcept`** 가 붙어있는지 꼭 확인합니다.
3.  만약 `noexcept`가 없다면? 벡터는 성능을 포기하더라도 안전을 위해 **'느린 복사(Copy)'**를 선택해 버립니다.
4.  결론적으로, 이동 시맨틱을 제대로 활용하려면 **`noexcept`** 는 선택이 아닌 필수입니다.

### Q5. "std::move()를 쓰면 원본 객체의 값이 자동으로 nullptr로 비워지나요?"
**아닙니다! 절대 아닙니다.** `std::move()` 자체는 단 1비트의 데이터도 건드리지 않습니다. 단순히 컴파일러에게 "얘를 R-value처럼 취급해 줘"라고 알려주는 **형변환(캐스팅) 스티커**일 뿐입니다. 
값이 `nullptr`로 비워지는 마법은, 그 스티커를 보고 호출된 **클래스의 이동 생성자(또는 이동 대입 연산자) 내부 로직**에서 작성자가 직접 비워주었기 때문에(ex. `other.ptr = nullptr;`) 일어나는 현상일 뿐입니다.

### Q6. "unique_ptr이 이동할 때, 그 안에 들어있는 실제 힙 객체(Weapon 등)의 이동 생성자도 함께 호출되나요?"
**호출되지 않습니다!** `unique_ptr`은 목줄이고, 실제 객체는 힙에 있는 강아지입니다. 
이동 시맨틱이 발생하면 오직 '목줄(포인터 주소)'의 소유권만 새 주인에게 넘어가고 예전 주인의 목줄을 끊어버릴 뿐입니다. 힙에 있는 실제 객체의 번지수나 데이터는 전혀 움직이지 않므로, 객체 본인의 이동 생성자가 불필요하게 호출되는 일은 없습니다. (이것이 포인터 이동의 압도적인 성능 비결입니다)

### Q7. "그러면 push_back 같은 함수는 여러 개가 오버로딩 되어 있는 건가요?"
**정확합니다!** C++11 이후 STL의 거의 모든 함수는 두 버전으로 오버로딩 되었습니다.
1. `push_back(const T&)`: L-value가 들어오면 무거운 **복사(Copy)**를 수행합니다.
2. `push_back(T&&)`: `std::move` 캐스팅이 붙은 R-value가 들어오면 이 버전을 낚아채서 가벼운 **이동(Move)**을 수행합니다. 

### Q8. "push_back(T&& t) 내부에서 또 std::move(t)를 한다고? 이미 R-value 스티커가 붙어있는데 왜 또 붙여요?"
**미친 폼의 질문입니다!** C++의 가장 유명한 함정(Gotcha)인 **"이름이 있는 변수는 무조건 L-value이다"**라는 절대 규칙 때문입니다.
함수의 매개변수 `t`의 타입이 `unique_ptr&&`(R-value 참조)인 것은 맞지만, **함수 내부로 들어오는 순간 `t`라는 '이름'이 생겨버립니다.** 
이름이 생겼다는 건, 우리가 `t`를 여러 번 반복해서 부를 수 있다는 뜻이죠. 즉, 함수 내부에서는 `t`가 다시 **L-value(수명 연장)로 둔갑**합니다!
만약 `push_back` 내부에서 `new (p) T(t);` 라고 그냥 넘기면, 컴파일러는 `t`를 L-value로 보고 **복사 생성자**를 부르려다 에러를 냅니다. 따라서 "얘가 아까 밖에서 훔쳐 오라고 허락받은 놈이야!"라고 다시 한번 `std::move(t)` (또는 템플릿의 경우 `std::forward`)를 씌워주어 R-value 스티커를 재부착해야만 무사히 이동 생성자로 넘어갈 수 있습니다. (스티커는 이름이 붙는 순간 떨어집니다!)

### Q9. "어차피 내부에서 L-value가 돼버리고 std::move를 또 써야 한다면, 처음부터 매개변수 타입으로 쓰인 `&&`의 존재 의미가 대체 뭡니까?"
`&&`(R-value 참조 매개변수)는 **문지기(Gatekeeper)**이자 **계약서** 역할을 합니다.
만약 `push_back(unique_ptr& t)` 라고 선언했다면, L-value만 들어올 수 있으므로 밖에서 임시 객체나 `std::move()`를 써서 던진 객체는 아예 함수 안으로 들어오지도 못합니다 (컴파일 에러).
반대로 `push_back(const unique_ptr& t)` 라고 선언했다면, 임시 객체를 받을 수는 있지만 `const` 철갑이 둘러져 있어서 내부에서 `std::move()`로 훔쳐 올(수정할) 방법이 막혀버립니다.
**결국 "임시 객체나 훔쳐가도 좋다고 허락받은 객체(R-value)만 전용으로 골라 받아서, 내부에서 그것들의 장기를 합법적으로 적출(수정)할 수 있는 권한"**을 가진 유일한 매개변수 타입이 바로 `T&&`인 것입니다.

### Q10. "내부에서 굳이 `new(p)` (Placement New)를 해주는 이유가 뭐죠? 그냥 대입(`*p = t`)이나 일반 `new`를 쓰면 안 되나요?"
`std::vector`의 가혹한 성능 최적화 때문입니다.
- **일반 `new`를 쓰면 안 되는 이유:** 일반 `new T()`는 1) 운영체제에 메모리를 달라고 요청(OS Call)하고, 2) 그 위에 생성자를 부릅니다. 벡터는 매크로 단위로 미리 엄청난 빈 땅(`capacity`)을 OS로부터 받아둡니다. 요소를 추가할 때마다 OS를 호출하면 너무 느리기 때문입니다. 우리는 새로운 땅이 필요한 게 아니라, **이미 있는 빈 땅 위에 건물(생성자)만 지어야** 합니다.
- **대입(`*p = t`)을 쓰면 안 되는 이유:** 벡터가 미리 잡아둔 빈 땅(p) 안쪽은 쓰레기값만 들어있는 **초기화되지 않은 날것의 메모리**입니다. 온전한 객체가 아니죠. 생성이 안 된 빈 땅에 대고 다짜고짜 대입 연산자(`operator=`)를 호출하면 프로그램이 터져버립니다. 무조건 '생성자'를 먼저 불러서 건물을 세워야 합니다.
- **해결책 (`new(p)`):** 그래서 C++은 "기존에 미리 잡아둔 빈 메모리 주소 p로 찾아가서, 다른 거 하지 말고 딱 **객체 생성자만 그 위치에서 실행**해라!"는 특수 문법인 Placement New (`new (주소) 생성자`)를 만들었습니다.

### Q11. "근데 R-value는 애초에 주소가 없는(메모리에 존재하지 않는) 임시 값 아니었나요? 주소도 없는데 어떻게 훔쳐 오고 포인터를 수정하나요?"
이 질문은 **C++11 Value Category**의 가장 깊은 바닥입니다! 
예전 기초 과정에서 "R-value는 `5` 처럼 이름도, 주소도 없다"고 배웠습니다. (물론 `&5`처럼 노골적으로 주소를 취할 순 없습니다.) 하지만 C++11부터는 이동 시맨틱을 위해 R-value의 개념이 두 가지로 쪼개졌습니다.
1. **prvalue(순수 R-value)**: `5`나 연산의 결과. 원래 주소가 없는 녀석들.
2. **xvalue(곧 소멸할 값 - eXpiring Value)**: **원래는 L-value(주소 있음)였지만, `std::move()`를 통해 "나 죽일 테면 죽여라" 하고 사망 선고를 받은 녀석들.** 

우리가 이동 시맨틱에서 훔쳐 오는 R-value들은 대부분 **xvalue**입니다! 당연히 원래는 주소가 있던 놈들이죠.
게다가 만약 진짜 임시 객체(prvalue)를 `&&`로 받게 되더라도 놀라운 마법이 일어납니다. C++은 R-value 참조(`&&`)가 임시 객체를 가리키는 순간, **스택의 보이지 않는 곳에 임시 객체의 몸(메모리)을 몰래 구체화(Materialize)**시켜 주고 수명을 연장해 줍니다!
즉, 매개변수 `t`가 받는 순간부터 그 R-value는 훔쳐 갈 수 있는 **분명한 주소(몸뚱아리)**를 갖춘 상태가 됩니다.

---

### Q12. "AssetVariant v1 = std::move(v2); 는 이동 대입 연산자가 호출되나요?"
**아닙니다!** 이것은 C++의 가장 흔한 오해 중 하나인 **'초기화(Initialization)'**와 **'대입(Assignment)'**의 차이입니다.
- **초기화 (`T v1 = rvalue;`)**: 새로운 객체가 탄생하는 순간이므로 **이동 생성자**가 호출됩니다. 실체는 `T v1(std::move(v2));`와 같습니다.
- **대입 (`v1 = rvalue;`)**: 이미 존재하던 객체에 값을 덮어쓰는 것이므로 **이동 대입 연산자**가 호출됩니다.

### Q13. "map의 `cache[path] = std::move(val);` 에서는 왜 map이 아니라 Value의 이동 대입이 호출되나요?"
*   **A**: 매우 예리한 관찰입니다! 
1. `cache[path]` 가 실행되면, 맵은 해당 키가 없으면 빈 공간을 먼저 만들고(Default Construction), 그 공간의 **참조자(`Value&`)**를 반환합니다.
2. 우리는 그 반환받은 **참조자**에 대고 `=` 연산을 수행합니다.
3. 결과적으로 `참조자(L-value) = R-value` 형태가 되어, 맵 전체가 아닌 **해당 칸에 들어있는 데이터(Value)의 이동 대입 연산자**가 발동되는 것입니다.

### Q14. "왜 `std::string&`는 임시 객체(R-value)를 못 받는데, `const std::string&`는 가능한가요?"
*   **A**: **안전성과 약속**의 차이입니다.
    *   **`T&` (비-상수 참조)**: "나는 원본을 **수정**할 거야!"라고 선언한 것입니다. 하지만 곧 사라질 임시 객체(R-value)를 수정하는 것은 논리적으로 모순이며 버그일 확률이 높기 때문에 컴파일러가 원천 차단합니다.
    *   **`const T&` (상수 참조)**: "나는 **읽기만** 할게!"라는 약속입니다. C++은 이 약속을 믿고, 임시 객체의 수명을 함수 호출이 끝날 때까지 **강제로 연장(Life-time Extension)**해 주어 안전하게 사용할 수 있도록 허용합니다.
