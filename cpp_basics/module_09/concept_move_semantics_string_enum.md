# Module 9 보충: std::string과 Enum의 이동 시맨틱(Move Semantics)

## 📌 모든 객체가 `std::move`의 혜택을 받는 것은 아니다!

복사/이동 생성자를 구현할 때 무거운 것을 다룰 때를 대비해 `std::move()`로 다른 객체가 가진 소유권을 날치기(?) 해오는 패턴은 아주 흔합니다. 하지만 데이터 타입의 근본 구조에 따라 `std::move`의 내부 이득(원리)은 극명하게 갈립니다.

### 1. `std::string`의 극강의 소유권 이동 (소유권 탈취 마술)
문자열 길이가 아주 길면 (예: 100만 자) `std::string`은 메모리의 느린 힙(Heap) 영역에 수백 메가바이트의 거대한 메모리 블록을 할당하여 글자를 저장합니다.
*   **복사 생성/대입 (`=` 방식)**: 똑같은 크기의 수백 메가바이트짜리 힙 메모리를 새롭게 할당받아, 원본의 100만 자를 한 글자씩 CPU가 모조리 베껴 씁니다. (매우 무겁고 느립니다.)
*   **이동 생성/대입 (`std::move()`)**: 원본 `std::string`이 꽉 쥐고 있던 **초거대 힙 메모리의 주소(포인터)**만 쏙 빼와서 내 명의로 바꾸고, 원본 객체에게는 빈 깡통(`nullptr`나 `""`)을 쥐어줍니다. (글자를 베끼지 않으니 0.0001초도 안 걸립니다!)

따라서 멤버 변수로 힙 메모리를 관리하는 타입(`std::string`, `std::vector`, `std::unique_ptr` 등)은 `std::move()`의 혜택을 드라마틱하게 누립니다.

### 2. `enum class`나 `int`에서의 `std::move`?
캐릭터의 상태나 아이템 등급을 나타내기 위해 만든 `enum class` 타입 같은 기본 데이터형(Primitive Types)들은 어떨까요?
```cpp
enum class ItemRarity { Common, Rare, Epic };

void UpdateRarity(ItemRarity&& rarity) {
    this->m_rarity = std::move(rarity); 
}
```
위 코드는 문법적으로 에러가 나지도 않고 아무 문제 없이 작동하지만, **이곳에서 `std::move`의 성능적 혜택은 사실상 0입니다.**
*   **이유**: 정수형(`int`, `enum class`), 주소 껍데기(`int*`), 문자(`char`) 같은 기본 타입들은 무거운 힙 메모리와 연결되지 않고 메모리(Stack 또는 레지스터)에 그 값 자체가 고스란히 적혀 있습니다. 
*   `std::move`를 씌워 우변값(rvalue) 캐스팅을 하더라도, C++ 컴파일러는 빼앗을 힙 포인터가 없으니 결국 안전하게 그 값 자체를 원래대로 베껴 쓰는(Copy) CPU 명령을 내립니다.

### 3. 결론 가이드라인
*   무거운 자원(Heap)이나 외부에 연결된 핸들을 들고 있는 클래스(`string/vector/unique_ptr`): **`std::move`의 소유권 이전 마법이 압도적 차이를 냄 (필수 고려 대상!!)**
*   가벼운 원시 구조체, `enum`, `int`: 굳이 `move`를 안 써도 상관없지만, 썼다고 해서 프로그램이 느려지는 페널티도 없음 (어차피 복사와 동일하게 작동). 템플릿 프로그래밍 등에서 코드의 일관성을 위해 모든 곳에 썼다고 해도 아무런 해가 되지 않습니다.
