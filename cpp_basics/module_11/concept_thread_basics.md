# Module 11: 병렬 프로그래밍 (Concurrency) - 기초

## 🎮 실무 프로젝트 상황 (샌드위치 교육법)
학습자님이 거대한 오픈월드 RPG를 개발하고 있다고 가정해봅시다. 
플레이어가 새로운 지역으로 진입할 때, 수 기가바이트의 텍스트 데이터와 리소스를 불러와야 합니다.

*   **문제**: 만약 이 로딩 작업을 게임의 화면을 그리는 메인 루프(Main Thread)에서 실행한다면?
*   **결과**: 로딩이 끝날 때까지 게임 화면은 멈추고(Freezing), 사용자는 게임이 튕긴 것으로 오해할 것입니다.

이것이 바로 우리가 **"병렬성(Concurrency)"**을 배워야 하는 이유입니다. 메인 스레드는 화면을 계속 그리게 두고, 다른 스레드에게 무거운 작업을 맡겨야 합니다.

---

## 🧵 스레드(Thread)란 무엇인가?

컴퓨팅에서 스레드는 **"프로세스 내에서 실행되는 하나의 흐름"**입니다.
기본적으로 모든 C++ 프로그램은 `main()` 함수에서 시작하는 하나의 **메인 스레드**를 가집니다.

### 1. `std::thread` 제어하기
C++11부터 표준 라이브러리(`#include <thread>`)를 통해 OS에 상관없이 스레드를 생성할 수 있게 되었습니다.

#### 기본 사용법:
```cpp
#include <iostream>
#include <thread>

void WorkerTask() {
    std::cout << "별도의 스레드에서 돌아가는 작업입니다.\n";
}

int main() {
    // 1. 스레드 생성 (WorkerTask 함수를 별도 흐름으로 실행)
    std::thread t1(WorkerTask);

    // 2. 관리 (둘 중 하나는 반드시 호출해야 합니다)
    t1.join();   // t1이 끝날 때까지 메인 스레드가 여기서 대기한다.
    // t1.detach(); // t1을 자유로운 영혼으로 풀어준다. (백그라운드 실행)

    return 0;
}
```

## 🚦 생명 주기와 하드웨어 시각화

스레드는 생성되는 순간 운영체제가 관리하는 별도의 실행 컨텍스트(Stack, PC 등)를 가집니다. 이를 눈으로 확인해봅시다.

### 스레드 식별 (ID)
```cpp
std::cout << "Main Thread ID: " << std::this_thread::get_id() << "\n";
std::thread t([](){
    std::cout << "Worker Thread ID: " << std::this_thread::get_id() << "\n";
});
t.join();
```
각 스레드는 고유한 ID를 가지며, 이는 하드웨어 레벨에서 서로 다른 흐름임을 증명합니다.

### 2. 스레드 인자 전달과 `std::ref` (Argument Passing)

스레드에 함수 인자를 전달할 때는 일반적인 함수 호출과는 다른 **"배달 원리"**가 적용됩니다.

#### 📦 배달부(std::thread)의 복사 전략
`std::thread` 생성자는 전달받은 모든 인자를 일단 **자기 내부 주머니에 복사(Decay Copy)**합니다.
*   **이유**: 새로운 스레드가 언제 실행될지 모르기 때문에, 메인 스레드에 있는 지역 변수가 사라지더라도 안전하게 작업할 수 있도록 **"자체 복사본"**을 챙기는 것입니다.

#### ❌ 문제 발생: 참조형(`&`) 파라미터와의 충돌
만약 함수가 `void Func(int& a)`처럼 참조를 받는다면, 다음과 같은 상황이 벌어집니다.
1.  배달부(Thread)가 인자를 가져가서 **자기만의 복사본(rvalue)**을 만듭니다.
2.  배달부가 목적지에 가서 함수에게 자기가 만든 **복사본**을 던져줍니다.
3.  함수는 **"난 원본(lvalue)만 받기로 약속했는데, 왜 복사본(rvalue)을 줘?"** 라며 거절합니다.
4.  결과: **컴파일 에러** (`std::thread arguments must be invocable after conversion to rvalues`)

#### 🛡️ 해결책: `std::ref` (Reference Wrapper)
이 성질을 극복하기 위해 `<functional>` 헤더의 `std::ref`를 사용합니다.
*   **원리**: `std::ref`는 원본을 가리키는 **"명함/약도(reference_wrapper)"**를 만듭니다. 
*   배달부는 이 **명함**을 복사해서 가져가고, 함수에 도착했을 때 그 명함에 적힌 주소를 보고 **진짜 원본**을 찾아가게 됩니다.

```cpp
void Increment(int& a) { a++; }

int main() {
    int count = 0;
    // std::thread t1(Increment, count); // ❌ 에러: 배달부가 복사본을 주려 함
    std::thread t1(Increment, std::ref(count)); // ✅ 성공: 명함을 복사해서 원본에 연결
    t1.join();
}
```

## 🔍 Q&A: 깊이 있는 이해

### 1. `detach()` 시 자원 해제가 안 되는 케이스는 무엇인가요?
`detach()`는 스레드 객체와 실제 실행 흐름 사이의 연결을 끊는 것입니다. 여기서 발생하는 위험은 다음과 같습니다:

*   **메인 함수 종료 (Process Exit)**: 메인 스레드가 `return 0`으로 종료되면, 운영체제는 해당 프로세스의 모든 자원을 즉시 회수합니다. 이때 실행 중이던 `detach` 스레드들은 소멸자가 호출될 기회도 없이 강제 종료됩니다. 
    *   만약 스레드가 파일에 데이터를 쓰고 있었다면 파일이 깨질 수 있고, 힙 메모리에 큰 데이터를 할당 중이었다면 비정상적인 상태로 프로세스가 끝날 수 있습니다.
*   **지역 변수 참조(Dangling Reference)**: 메인 함수의 지역 변수 주소를 `detach` 스레드에 넘겼는데, 메인 함수가 먼저 끝나버리면 해당 스레드는 이미 사라진 스택 메모리를 참조하게 됩니다. 이는 매우 위험한 **정의되지 않은 동작(Undefined Behavior)**입니다.

### 2. 스레드끼리 공유하는 것 vs 공유하지 않는 것

스레드는 같은 프로세스라는 "집"을 공유하지만, 각자의 "작업 가방"은 따로 가지고 있습니다.

*   **공유하는 것 (Shared)**:
    *   **힙(Heap) 영역**: `new`로 할당한 메모리는 모든 스레드가 접근 가능합니다.
    *   **코드(Code) 영역**: 같은 실행 코드를 공유합니다.
    *   **데이터(Data) 영역**: 전역 변수나 정적(`static`) 변수는 공유됩니다. (데이터 오염의 주범!)
    *   **파일/소켓**: 열려 있는 파일 핸들이나 네트워크 연결은 공유합니다.
*   **공유하지 않는 것 (Private)**:
    *   **스택(Stack)**: 함수 호출 이력과 지역 변수는 각 스레드마다 독립적인 메모리 공간을 가집니다. (매우 중요!)
    *   **레지스터(PC, SP 등)**: 현재 어떤 코드를 읽고 있는지(Program Counter), 스택 어디를 가리키는지(Stack Pointer)는 CPU 레벨에서 각자 관리됩니다.
    *   **TLS (Thread Local Storage)**: 스레드 전용 전역 저장 공간.

## 🚫 주의사항: Join과 Detach의 운명
*   **`join()`**: 자식이 집에 올 때까지 부모가 현관문 앞에서 기다리는 것과 같습니다. 자원 해제가 안전하게 보장됩니다.
*   **`detach()`**: 자식을 호적에서 파는 것과 같습니다. 부모는 자식이 언제 죽는지(끝나는지) 알 수 없으며, 프로그램 종료 시 자원이 제대로 해제되지 않을 위험이 있습니다.
*   **필수 원칙**: `std::thread` 객체가 소멸되기 전, 반드시 `join` 또는 `detach` 중 하나를 결정해야 합니다. 결정하지 않고 객체가 소멸되면 **런타임 에러(abort)**가 발생합니다.

## 🏗️ 데이터 흐름 및 소유권 정리 (Ownership Summary)

오늘 학습을 통해 정리한 **"스레드 인자 전달 시나리오"**의 최종판입니다.

| 전달 방식 | 배달부(`std::thread`)의 동작 | 받는 함수(`Increment`)의 행동 | 결과 및 용도 |
| :--- | :--- | :--- | :--- |
| **`int a`** (값) | 주머니의 복사본을 **이동(move)**하여 던짐 | 던진 걸 받아서 **새로운 공간(`a`)**에 기록함 | **복사본의 복사본** 생성. (가장 안전하지만 무거움) |
| **`int &&a`** (rvalue 참조) | 주머니의 복사본에 대한 **소유권을 포기**함 | 버려진 데이터 자리에 **내 이름표(`a`)만 붙임** | **제로 카피(Zero-copy)**. 스레드 내부 데이터 재사용. |
| **`int &a` + `std::ref`** | 메인에 있는 **원본 주소(명함)**를 복사해줌 | 명함을 보고 **메인 스레드의 원본**에 빨대를 꽂음 | **원본 수정 가능**. 우리가 오늘 실습에서 사용한 방식. |
