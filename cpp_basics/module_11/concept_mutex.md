# Module 11: 병렬 프로그래밍 (Concurrency) - Race Condition과 std::mutex

## 🎮 실무 프로젝트 상황 (샌드위치 교육법)
여러분이 MMORPG 서버를 개발 중이라고 가정해 봅시다. 
골드(Money)를 관리하는 중앙 은행 계좌가 있습니다. 
광전사와 마법사가 동시에 사냥을 마치고 "내 계좌에 100 골드를 추가해줘!" 라고 서버(공유 자원)에 요청했습니다.

*   **정상적인 기대**: 0골드 -> 100골드 추가 -> 200골드 추가. (최종 200골드)
*   **실제 벌어지는 일**: 둘 다 동시에 접근하면 최종 골드가 **100골드**로 덮어씌워지는 대참사(골드 복사 또는 증발)가 발생할 수 있습니다.

이처럼 여러 스레드가 하나의 공유 자원에 동시에 접근하고 수정하려 할 때 데이터가 엉키는 현상을 **경쟁 상태(Race Condition)**라고 합니다.

---

## 💥 경쟁 상태(Race Condition)는 왜 발생할까?

C++ 코드 상에서 `money++` 라는 아주 간단해 보이는 한 줄은, 실제 CPU 하드웨어 레벨(어셈블리)에서는 **3개의 독립된 동작**으로 쪼개어 실행됩니다.

1.  **Read**: 메모리에 있는 현재 `money` 값을 CPU Register(작업대)로 가져온다.
2.  **Modify**: Register에서 100을 더한다.
3.  **Write**: 계산된 결과를 다시 메모리의 `money` 변수에 덮어쓴다.

만약 광전사 스레드와 마법사 스레드가 **동시에 1번 동작(Read)을 해버렸다면?**
둘 다 0골드를 읽어가서 각자 100골드를 더한 후, 같은 자리에 100골드를 덮어쓰게 됩니다. 결과적으로 누군가의 돈은 증발해 버립니다.

---

## 🛡️ 해결책: 락(Lock)을 건다, `std::mutex`

화장실(공유 자원)에 들어갈 때 문을 잠가서 다른 사람(스레드)이 들어오지 못하게 막는 것과 같은 원리입니다.
C++에서는 이를 `<mutex>` (상호 배제, Mutual Exclusion의 약자)로 제공합니다.

### 어떻게 동작하는가?
*   `mutex.lock()`: 화장실 문을 잠급니다. 만약 이미 잠겨있다면, 열릴 때까지 (CPU를 내려놓고) 밖에서 무한 대기(Block)합니다.
*   `mutex.unlock()`: 화장실 문을 엽니다. 비로소 대기하던 다음 스레드가 진입할 수 있습니다.

```cpp
#include <iostream>
#include <thread>
#include <vector>
#include <mutex>

int shared_money = 0; // 공유 자원 (모든 스레드가 접근)
std::mutex m_lock;    // 문을 걸어 잠글 자물쇠 객체

void AddMoney() {
    for (int i = 0; i < 10000; i++) {
        // [임계 구역 (Critical Section) 진입 전]
        m_lock.lock();   // 내 작업이 끝날때까지 아무도 건들지 마!
        
        shared_money++;  // 자원 수정 (안전하게 1마리씩만 실행됨)
        
        m_lock.unlock(); // 작업 끝! 다음 사람 들어와도 좋아.
        // [임계 구역 (Critical Section) 탈출]
    }
}
```

---

## 👁️ 하드웨어 검증: 락(Lock)이 없다면?

만약 위 코드에서 `m_lock.lock()`과 `unlock()`을 지운 상태로 수십 개의 스레드를 동시에 돌리면 어떻게 될까요?
`sizeof`나 `&` 주소 출력처럼 직관적이지 않지만, 이 **충돌 확률** 자체가 하드웨어 동기화 부재의 증거가 됩니다. (1만 번을 반복시켜야 CPU 타이밍이 꼬이는 것을 볼 수 있기 때문에 반복문을 돌립니다.)

*정확히 20000이 나오지 않고, 매번 14231, 16930 같이 예측할 수 없는 쓰레기값이 나온다면, 그것이 바로 Race Condition 카오스를 증명하는 것입니다.*

## 🚨 주의사항: 데드락 (Deadlock)과 예외 처리의 공포

수동으로 `mutex.lock()`을 한 상태에서, 내부 로직 중에 0으로 나누기 에러(Exception)가 나거나 `if (error) return;`으로 함수를 조기 이탈해 버리면 어떻게 될까요?
**`unlock()`이 평생 호출되지 않아, 문 밖에서 기다리던 다른 모든 스레드들이 영원히 멈추는 데드락(Deadlock, 교착 상태)**에 빠지게 됩니다. 

### 🛡️ 모던 C++의 해결책: `std::lock_guard` (RAII 기반 자동 자물쇠)
C++ 커뮤니티에서는 수동으로 `.lock()`과 `.unlock()`을 타이핑하는 것을 **강력하게 억제**합니다. 대신 스마트 포인터(`unique_ptr`)와 같은 원리를 사용하는 자물쇠 특수 관리 객체를 씁니다.

```cpp
#include <mutex>

int shared_money = 0;
std::mutex m_lock;

void AddMoneySafe() {
    // 락 가드 객체가 생성되는 순간 자동으로 m_lock.lock()을 호출합니다.
    std::lock_guard<std::mutex> guard(m_lock);
    
    shared_money++;
    
    // (이곳에서 치명적인 에러가 터져서 함수가 비정상 종료되어도 안전함)
    
} // 함수가 끝나는(중괄호 탈출) 순간, guard 변수가 스택에서 파괴되며 
  // 소멸자가 자동으로 m_lock.unlock()을 100% 보장하여 풀어줍니다!
```
이처럼 객체의 수명 주기(스코프)에 자원 해제를 의존하는 기법을 **RAII (Resource Acquisition Is Initialization)**라고 하며, C++에서 메모리 누수와 데드락을 방어하는 가장 핵심적인 철학입니다.

#### 💡 예상 질문: 예외가 터졌는데 어떻게 소멸자가 알아서 호출되나요? (Stack Unwinding)
"함수 중간에 심각한 에러(Exception)가 터져서 튕겨 나가버렸는데, 어떻게 `guard` 변수가 잊지 않고 파괴될까요?"

이것은 C++ 언어 자체가 하드웨어 레벨에서 강력하게 보장하는 **스택 풀기(Stack Unwinding)** 메커니즘 덕분입니다. 
C++ 런타임은 예외가 발생해서 스코프(`{ }`)를 비정상적으로 찢고 탈출할 때조차도, **그 스코프 안에서 만들어졌던 모든 '지역 변수(Stack 메모리)'들을 하나하나 거꾸로 찾아가며 반드시 소멸자(Destructor)를 호출해 주고** 떠납니다. 
따라서 에러가 나서 튕겨 나갈 때, 언어 차원의 뒷정리 시스템이 `guard`의 소멸자를 기계적으로 실행시키고, 그 소멸자 안에 든 `m_lock.unlock()`이 무조건 발동되면서 데드락을 완벽하게 막아내는 것입니다.

#### 🚨 번뜩이는 통찰: 락가드는 왜 굳이 '지역 변수'로 선언해야 할까?
"메인 객체(`Bank`) 안에 있는 `mutex`는 함수 밖에서 안전하게 살아있는데, 함수 내부에서 에러가 터졌다고 왜 락이 풀리지 않은 채 영원히 방치될까요?"
정답은 `mutex` 자체가 **스택 풀기(청소) 대상에 포함되지 않기 때문**입니다! 함수에서 에러가 나면 런타임은 **함수 내부의 지역 스택 변수들만 청소**하고 급하게 탈출합니다. 생명주기가 함수 밖(`Bank`)에 있는 진짜 `mutex`는 청소 대상이 아니므로 잠긴 채로 영원히 버려집니다(Deadlock). 

바로 이 지점이 **`std::lock_guard`를 함수 내부의 '지역 변수'로 선언해야 하는 궁극의 이유**입니다. 
지역 스택 메모리에 대리인(`guard`)을 만들어 두면, 함수가 비정상적으로 터져 스택이 청소될 때 이 대리인도 함께 파괴되며, 그 대리인의 마지막 유언(소멸자)이 **"멀리 있는 진짜 자물쇠를 대신 `unlock()` 해라!"** 로 동작하여 서버 전체가 멈추는 재앙을 막아냅니다.

## 🧩 다양한 종류의 자물쇠들 (Other Types of Locks)

질문하신 것처럼, C++에는 단순한 `std::mutex` 외에도 상황에 맞게 쓸 수 있는 최적화된 락(Lock) 메커니즘이 존재합니다. MMORPG 게임 서버를 계속 예로 들어볼까요?

### 1. `std::shared_mutex` (C++17) - 읽기/쓰기 락 (Read-Write Lock)
**상황**: 아이템 백과사전의 정보를 **조회(Read)**하는 유저는 수천 명인데, 운영자가 가끔 정보를 **수정(Write)**한다고 해봅시다. 단순 `mutex`를 걸면 데이터를 건드리지 않는 조회 유저들끼리도 서로를 기다려야 해서 서버 성능이 끔찍하게 저하됩니다.

*   **읽기 전용 래퍼 (`shared_lock` == DB의 S-Lock)**: 데이터를 수정하지 않고 읽기만 할 때 사용합니다. 세마포어처럼 "최대 100명" 같은 인원 제한이 따로 있는 것이 아니라, **읽기 요청이라면 무제한으로 동시 입장을 허용**합니다.
    *   *RAII 지원*: `lock_guard`와 완전히 똑같습니다. 생성 즉시 락을 걸고, 스코프(`{ }`)를 벗어나면 알아서 락을 풉니다.
*   **쓰기 전용 래퍼 (`unique_lock` == DB의 X-Lock)**: 데이터를 오염시킬 위험이 있는 **수정 작업** 시에만 사용합니다. 방 안을 읽는 유저가 **정확히 0명**이 될 때까지 대기한 후, 완전히 빈 방이 되면 단 혼자만 들어가서 자원을 독점합니다.
    *   *RAII 지원*: 역시 범위를 벗어나면 스스로 락을 푸는 스마트 래퍼입니다.
*   🛡️ **심화: 운영자의 기아 상태(Writer Starvation) 방지**: 무제한으로 유저가 들어올 수 있다면, 읽는 유저가 끝없이 쏟아질 때 운영자는 평생 쓰기(Write)를 못하고 굶어죽는(Starvation) 것 아닐까요? 
    *   C++ 시스템(OS에 따라 동작 방식은 약간 다를 수 있으나, 일반적으로 C++14 `shared_timed_mutex`나 C++17 `shared_mutex` 구현체)은 아주 똑똑합니다. 운영자가 밖에서 **"나 이제 들어가서 쓸 거야!" 하고 대기열에 서는 순간, 눈치껏 추가적인 읽기 유저(Reader)의 신규 진입을 셔터 내리듯 막아버립니다.** 그리고 방 안에 남아있던 기존 유저들이 다 빠져나가서 빈 방이 되면, 운영자가 무사히 들어가 쓰기 락을 거머쥐게 됩니다.

**💻 실무 코드 예시:**
```cpp
#include <iostream>
#include <shared_mutex>
#include <string>

class ItemDictionary {
    std::string item_data = "초기 방어구 통계 데이터";
    std::shared_mutex rw_lock; // 본체 자물쇠는 딱 1개만 존재합니다.

public:
    // [유저 1만 명이 동시에 호출 가능한 읽기 함수]
    std::string ReadData() {
        // 읽기 전용 껍데기를 씌웁니다. 
        // 쓰기 중인 운영자만 없다면 무제한으로 통과! (병목 현상 없음)
        std::shared_lock<std::shared_mutex> guard(rw_lock); 
        return item_data; 
    }

    // [운영자가 가끔 호출하는 쓰기 함수]
    void WriteData(const std::string& new_data) {
        // 쓰기 전용(독점) 껍데기를 씌웁니다. 
        // 현재 읽고 있는 유저가 0명이 될 때까지 기다렸다가, 나 혼자 독점!
        std::unique_lock<std::shared_mutex> guard(rw_lock); 
        item_data = new_data;
    }
};
```

### 2. `std::recursive_mutex` - 재귀적 락
**상황**: `Attack()` 함수 안에서 자물쇠(`lock()`)를 걸었는데, 그 안에서 또 다른 내부 함수 `CalculateDamage()`를 불렀고 거기서 **같은 자물쇠에 한 번 더 `lock()`을 시도**한다면 어떻게 될까요? 단순 `mutex`는 자기가 잠가놓고는 자기가 문 밖에서 영원히 기다리는 바보 같은 데드락(Self-Deadlock)에 빠집니다.
**특징**: 같은 스레드(같은 사람)라면 자물쇠를 2번 3번 중복해서 걸 수 있게 통과시켜 줍니다. (단, 풀 때도 무조건 잠근 횟수만큼 `unlock()`을 해야 합니다).
*   **주의**: 구조가 스파게티처럼 꼬일 위험이 높아 모던 엔지니어링에서는 **사용을 피하고 설계를 고치는 것을 강력히 권장**합니다.

### 3. 스핀락 (Spinlock) / 바쁜 대기(Busy Waiting)
**상황**: 화장실 문이 잠겨있습니다. 그런데 안에 있는 사람이 금방 손만 씻고 **0.001초 만에 나올 것이 확실**합니다.
일반 `mutex`는 문이 잠겨있으면 잠이 들어 버립니다(Sleep & OS Context Switch). 그런데, 이렇게 잠들고 다시 깨어나는 운영체제 개입 과정 자체가 문 앞에서 잠깐 기다리는 것보다 CPU 비용이 훨씬 비쌉니다.
**특징**: 자러 가지 않고, 문 앞에서 계속 멤돌면서 **"나왔어? 나왔어?" 하며 무한 루프를 돌며 확인합니다(Spin).** 
*   모던 C++20 이전에는 주로 `std::atomic_flag` 자료형을 활용하여 구현했으며, 아주 짧고 극한의 속도가 필요한 핵심 코어 로직(예: 메모리 할당기)에서 필수적으로 사용됩니다.
*   💡 **심화 (`yield`의 활용)**: 스핀락처럼 의미 없이 무한 루프를 돌 때 CPU 100% 점유 현상을 막기 위해, 루프 내부에 **`std::this_thread::yield()`** 함수를 넣을 수 있습니다. 이는 OS 스케줄러에게 *"저 지금 할 일 없으니 남은 CPU 할당 시간(Timeslice)을 양보하고 대기열 뒤로 갈게요. 다른 준비된 스레드부터 싱글/멀티 코어에서 실행시켜 주세요"* 라고 통보하는 문맥 교환(Context Switching) 유도 테크닉입니다.

### 4. 세마포어 (Semaphore) - C++20 도입
**상황**: 화장실이 하나가 아니라 **칸이 3개 있는 공중화장실**입니다. 문 잠그는 자물쇠(`mutex`)로는 "딱 1명만" 통제가 가능하지만, 이 화장실은 칸이 남으면 3명까지는 동시에 들어갈 수 있게 해줘야 합니다.
**특징**: `mutex`가 0과 1 상태(잠금/풀림)만 갖는다면, 세마포어는 **'카운터(숫자)'**를 갖습니다. 3명이 들어가면 카운터가 0이 되어 다음 사람은 대기하고, 한 명이 나오면 카운터가 1이 되어 대기하던 사람이 들어갑니다.
*   `std::counting_semaphore<N>`: 초기 카운트를 N개로 두고 시작하는 형태입니다. (스레드 풀, 커넥션 풀 크기 제한에 사용)

#### 💡 심화: `std::binary_semaphore` (최대 카운트 1) vs `std::mutex`의 결정적 차이
두 객체 모두 최대 1명만 들여보낸다는 결과는 같습니다. 하지만 내부 **아키텍처 철학**이 완전히 다릅니다.

1.  **소유권 (Ownership)의 유무**
    *   **Mutex**: 소유권이 **있습니다**. 화장실 문을 A 스레드가 잠갔다면, **무조건 A 스레드가 안에서 열고 나와야 합니다**. A가 잠갔는데 B가 밖에서 억지로 열어버리면 C++에서는 치명적인 에러(Undefined Behavior)가 발생합니다. (철저한 **자원 보호** 목적)
    *   **Binary Semaphore**: 소유권이 **없습니다**. 이것은 자물쇠가 아니라 단순히 '0과 1'을 가리키는 **신호등**입니다. A 스레드가 신호등을 빨간불(0)로 만들고 자러 가버렸을 때, **B 스레드가 대신 신호등을 파란불(1)로 바꿔서 A를 깨워줄 수 있습니다(Signaling)**. (A가 요청을 보내면 B가 처리 후 '다 됐어!' 하고 원격으로 알려주는 **스레드 간 통신**에 특화됨)

2.  **OS 최적화 (Priority Inheritance - 우선순위 역전 방지)**
    *   **Mutex**: 운영체제가 "아, 이 자물쇠 안에는 지금 중요한 스레드가 갇혀 있구나"를 인지하고, 자물쇠를 거머쥔 느린 스레드를 일시적으로 빨리 달릴 수 있게 부스트업 시켜주는 스마트한 기능이 OS 차원에 존재합니다.
    *   **Semaphore**: 소유자가 누군지 모르기 때문에 이런 스마트한 우선순위 조정 로직이 불가능합니다. 

*   **요약**: 내 데이터를 다른 스레드로부터 안전하게 **방어**하려면 **`mutex`**를 써야 하고, A 스레드와 B 스레드가 서로 **"준비 완료!" 신호를 주고받아야** 한다면 **`binary_semaphore`**를 써야 합니다.

## 🤔 깊은 고민: 신호를 주고받으면 직렬(Serial)이랑 똑같은 거 아닌가요? (Producer-Consumer)

"스레드 B가 스레드 A의 신호를 기다렸다가 실행된다면, 결국 순서대로 실행되는 거니까 직렬(Serial) 프로그램이랑 다를 게 없는 거 아닌가요? 이게 과연 성능에 이득이 있나요?"

이 질문은 멀티스레드의 정수인 **'파이프라인(Pipeline)'**과 **'생산자-소비자(Producer-Consumer) 패턴'** 의 핵심을 관통하는 엄청난 질문입니다.

결론부터 말씀드리면, **"신호를 기다리는 아주 짧은 순간을 제외한 모든 시간 동안 각자의 작업을 100% 동시에(병렬로) 처리하기 때문에 압도적으로 빠르다"**가 정답입니다.

### 🍔 햄버거 가게 비유 (직렬 vs 병렬)

*   **직렬 (Serial/1코어)**: 요리사(메인 스레드)가 패티를 굽습니다(5분). 다 구우면 자기가 직접 홀에 나가 손님에게 서빙합니다(5분). **햄버거 하나당 총 10분**이 걸리고, 홀에 나가 있는 동안 주방은 텅 빕니다.
*   **병렬 (멀티스레드 + 신호등)**: 
    *   **요리사 스레드**: 패티 굽기에만 집중(5분). 구워진 패티를 테이블에 올리고 **종(Semaphore Signal)을 칩니다.** 그리고 **쉬지 않고 0.001초 만에 바로 다음 패티를 굽기 시작**합니다.
    *   **서빙 스레드**: 평소엔 스마트폰을 보며 대기합니다(Wait). 종이 울리면 햄버거를 들고 나가 서빙합니다(5분). 

이 병렬 처리의 마법은 **"요리사가 2번째 패티를 굽고 있는 5분 동안, 서빙 스레드가 1번째 햄버거를 서빙하고 있다"**는 점에 있습니다. 서로 다른 CPU 코어에서 각자의 무거운 작업(굽기, 걷기)을 100% 동시에 수행하며, 단지 테이블에 햄버거를 올려둘 때(종 칠 때)만 잠깐 통신하는 것입니다.

### 💻 게임 서버 실무 예시
MMORPG에서 이런 구조는 밥 먹듯이 쓰입니다.
1.  **네트워크 스레드(수신자)**: 무선/유선 인터넷을 통해 들어온 수천 개의 잡다한 데이터 패킷의 압축을 풀고 암호를 해독합니다(무거운 작업). 해독이 끝나는 족족 Queue(대기열)에 넣고 **세마포어 신호(Signal)**를 보냅니다.
2.  **로직 스레드(처리자)**: 신호가 오면 Queue에서 데이터를 꺼내 '플레이어 데미지 반영, 사망 판정, 위치 이동' 등 핵심 메모리 연산을 수행합니다(무거운 작업).

이 둘이 직렬로 동작한다면, 패킷 암호를 푸는 동안 게임의 모든 물리 렌더링과 데미지 계산이 멈춰 렉(Lag)이 걸릴 것입니다. 세마포어(또는 `condition_variable`)는 이 두 거대한 공장 컨베이어 벨트를 부드럽게 이어주는 톱니바퀴 역할을 할 뿐, 스레드 전체를 직렬로 묶어버리는 것이 절대 아닙니다.

## 🔑 상상해보기: 스레드가 "열쇠"를 들고 방(임계 영역)에 들어가는 걸까?

"락을 걸면 스레드가 열쇠를 주머니에 넣고 방에 들어가는 건가요?" 

멀티스레드 프로그래밍을 할 때 절대 잊어선 안 되는 아주 중요한 **'멘탈 모델(Mental Model)'**에 대한 훌륭한 질문입니다. 정답은 "아닙니다"에 가깝습니다. 실상은 **'공용 화장실 문 앞의 사용 중(Occupied) 팻말'**에 훨씬 더 가깝습니다.

1.  **물리적인 결계가 아닙니다**: `mutex` 객체는 데이터 변수(`money`)에 물리적인 보호막을 씌우는 것이 아닙니다. 단지 메모리의 한켠에 놓인 **"표지판(상태 변수)"**일 뿐입니다.
2.  **원자적(Atomic) 팻말 돌리기**: 스레드 A가 `lock()`을 호출하면, CPU의 특수 명령어를 이용해 누구보다 빠르게 표지판을 `[비어있음] -> [스레드 A 사용 중]`으로 돌려놓고 임계 영역 안으로 들어갑니다. (열쇠를 가져가는 게 아니라 팻말에 자기 이름을 적는 것에 가깝습니다).
3.  **신사협정 (Gentleman's Agreement)**: 스레드 B가 들어오려 할 때 표지판을 보고 `[사용 중]`이면 멈춰 서서 잡니다. 여기서 충격적인 사실은, 만약 깡패 같은 스레드 C가 **`lock()`을 호출하지 않고 그냥 무단침입해서 `money++`를 해버리면 막을 방법이 없다는 점입니다!** 

즉, `mutex`는 코드들이 **"우리 들어가기 전에 무조건 저 표지판 먼저 확인하고 돌려놓고 들어가자"**라고 맺은 '신사협정'일 뿐입니다. 코드 구조를 짤 때 모든 경로에서 락을 제대로 걸어주는 철저한 설계(캡슐화)가 왜 중요한지 여기서 드러납니다.

## 🚫 치명적인 실수: "왜 지역 변수(Non-static)로 뮤텍스를 선언하면 안 되나요?"

종종 초보자분들이 함수 내부에서 락을 걸기 위해 실수로 아래와 같이 코드를 짜는 경우가 있습니다.

```cpp
void hunt() {
    std::mutex m; // 함수 내부의 지역 변수로 선언! (대참사)
    std::lock_guard<std::mutex> guard(m);
    money++;
}
```

이 코드는 문법적으로 아무 에러가 나지 않지만, **100% 동기화에 실패하고 데이터가 오염됩니다.** 이유가 무엇일까요?

*   **스레드와 스택(Stack)**: 이전 스레드 기초에서 배웠듯, 모든 스레드는 자기만의 독립적인 **스택(작업 가방)** 공간을 가집니다.
*   **각자 가져온 개인 자물쇠**: 함수 내부에 일반 지역 변수로 선언된 `mutex`는 각 스레드가 이 함수에 진입할 때마다 **자신의 스택 메모리 안에서 새롭게 만들어집니다.**
*   즉, 100명의 분신(스레드)이 1개의 공용 화장실(money)에 모였는데, **각자 집에서 자물쇠와 문을 하나씩 들고 와서 허공에 세우고 문을 잠근 뒤, "좋아 문 잠궜어!" 하고 화장실로 우르르 뛰어 들어가는 꼴**입니다. 당연히 모두 프리패스로 충돌하게 됩니다.

**[해결책]**
그래서 자물쇠(`mutex`)는 **반드시 모든 스레드가 바라볼 수 있는 유일한 '공용(Shared)' 공간**에 있어야 합니다.
1.  **클래스의 멤버 변수**: (우리가 실습에서 한 방식) `Bank`라는 단 하나의 은행 지상층에 자물쇠를 매달아 놓습니다.
2.  **`static` 지역 변수**: 아예 함수 내부에 `static std::mutex m;` 처럼 선언해서, 이 함수를 몇 명이 호출하든 메모리 데이터 영역에 박힌 딱 1개의 자물쇠만 공유하도록 만들어야 합니다.
3.  **전역 변수(Global)**: 잘 쓰지 않지만 원리는 위와 동일합니다.

### ✨ [Advanced] `mutable` 과 `const` 멤버 함수 (M11 심화)

일반적으로 `const` 멤버 함수 안에서는 객체의 데이터를 수정할 수 없습니다. 하지만 **뮤텍스(`std::mutex`)**는 예외적인 상황에 처합니다.

```cpp
class Bank {
    int money;
    mutable std::mutex m_lock; // mutable 선언!

public:
    int GetMoney() const {
        // 읽기 전용 함수지만, 멀티스레드 환경에서는 락이 필요함!
        // lock_guard가 뮤텍스를 '수정(잠금)'해야 하는데, 
        // mutable이 없으면 const 함수 내에서 잠글 수 없습니다.
        std::lock_guard<std::mutex> lock(m_lock); 
        return money;
    }
};
```

**왜 `mutable` 인가요?**
- 뮤텍스를 잠그고 푸는 로직은 객체의 **논리적인 상태(데이터)**를 바꾸는 것이 아닙니다. 
- 단지 데이터를 안전하게 읽기 위한 **부수적인 수단**일 뿐입니다.
- C++은 이런 경우를 위해 `const` 함수 내에서도 수정 가능한 `mutable` 키워드를 제공합니다.
