# Module 11: 병렬 프로그래밍 (Concurrency) - std::jthread (C++20)

## 🎮 실무 프로젝트 상황 (샌드위치 교육법)
우리는 앞선 실습에서 `std::lock_guard`를 통해 자물쇠가 스코프를 넘어가면 알아서 풀리는(RAII) 기적을 보았습니다.
하지만, 만약 락이 아니라 **스레드 그 자체**에서 에러가 터졌다면 어떨까요?

서버에서 유저의 행동 로그를 디스크에 저장하는 `BackgroundLogger` 스레드를 하나 돌리고 있다고 가정해 보겠습니다. 
그런데 메인 흐름(혹은 함수 내부)에서 갑자기 메모리 에러가 터져버렸습니다. C++은 스택 풀기(Stack Unwinding)를 하며 스레드 객체(`std::thread` 지역 변수)를 파괴하려 할 것입니다. 

*   **기대하는 동작**: 로그 저장 스레드가 안전하게 종료되고 메인 프로그램이 예외를 처리한다.
*   **실제 벌어지는 일**: `std::thread`의 소멸자는 자신이 아직 돌아가고 있는데(joinable) 누군가 자신을 파괴하려 하면, **그 즉시 `std::terminate()`를 호출하여 서버 전체를 폭파(Abort)시켜 버립니다.**

이 어처구니없는 설계 결함(조인 망각 시 강제 종료)을 해결하기 위해 무려 9년 만인 C++20에 등장한 영웅이 바로 **`std::jthread`**입니다.

---

## 🦸‍♂️ 구원자의 등장: `std::jthread` (Joining Thread)

`std::jthread`는 기존 `std::thread`와 사용법이 99% 똑같지만, 내부 설계 철학이 완전히 다릅니다. 바로 **RAII(생성 시 획득, 소멸 시 반환)** 원칙을 스레드 객체 스스로에게 완벽하게 적용한 녀석입니다.

### 1. 알아서 다가오는 작별인사 (Auto-Join)
`std::thread`는 우리가 손수 `join()`이나 `detach()`를 불러주지 않으면 화를 내며 프로그램을 터뜨렸습니다.
하지만 `std::jthread`는 스코프(`{ }`)를 벗어나거나 에러가 터져서 소멸자가 불리는 순간, **스스로 `join()`을 호출하여 해당 스레드의 작업이 안전하게 끝날 때까지 메인 흐름을 잠깐 대기시켜 줍니다.**

```cpp
#include <iostream>
#include <thread> // jthread 도 동일한 헤더 사용

void FileDownload() {
    std::cout << "[jthread] 백그라운드 다운로드 중...\n";
}

int main() {
    {
        std::jthread worker(FileDownload);
        // 여기서 함수가 끝날 때 우리가 worker.join()을 적지 않았지만,
        // jthread는 지역 변수가 파괴될 때 스스로 join()을 호출하고 조용히 기다려줍니다.
    }
    std::cout << "스코프를 벗어나면 안전하게 조인된 후 여기까지 도달합니다.\n";
    return 0;
}
```

## 🛑 우아한 중단 요청: `std::stop_token` (Cooperative Cancellation)

`std::jthread`의 진짜 킬러 기능은 사실 Auto-Join이 아니라 **'협력적 중단 요청'**입니다.

강제로 스레드의 목을 졸라서 죽이는 것(`TerminateThread` 같은 OS 함수)은 자원 누수로 직결되는 최악의 방법입니다. 따라서 스레드를 멈추려면 밖에서 "이제 그만 자러 갈 시간이야~" 라고 깃발을 흔들고, 스레드가 자기 작업을 마무리하다가 스스로 깃발을 보고 정리 과정을 거친 뒤 종료하는 것이 제일 안전합니다.

`std::jthread`는 이 기능을 기본 내장하고 있습니다.
*   `jthread.request_stop()`: 밖에서 "그만 멈춰줄래?" 라고 깃발을 듭니다.
*   `stop_token.stop_requested()`: 안에서 돌고 있는 작업자가 "누가 나보고 멈추라 했나?" 하고 확인합니다.

```cpp
// 매개변수로 std::stop_token을 받아야 밖에서 흔드는 깃발을 인지할 수 있습니다.
void EndlessTask(std::stop_token stoken) {
    while (true) {
        // 작업을 하다가 틈틈이 깃발이 올라왔는지 뒤를 돌아봄
        if (stoken.stop_requested()) {
            std::cout << "중단 요청 확인 완료. 열려있는 파일 닫고 안전하게 퇴근합니다.\n";
            return; // 스스로 우아하게 종료
        }
        std::cout << "일하는 중...\n";
    }
}

int main() {
    std::jthread worker(EndlessTask);
    
    // ... 메인 흐름에서 어떤 조건 충족 시 ...
    worker.request_stop(); // "이제 그만하고 퇴근해!"
    
    // jthread 소멸자가 불릴 때 알아서 join() 해 주므로 끝.
}
```

## 🏛️ 실무 아키텍처: 우아한 종료 (Graceful Shutdown)
방금 짚어주신 내용은 실제 백엔드 서버(게임 서버, 웹 서버 등)의 **Graceful Shutdown 아키텍처**와 정확하게 일치합니다! `stop_token`은 이 아키텍처를 구현하기 위한 C++의 궁극적인 도구입니다.

1. **별도의 스코프(`{ }`)가 없다면?**: 맞습니다. `main` 함수에서 스코프 없이 곧바로 `jthread`를 만들었다면, `main`의 끝 지점(return 0 직전)에 도달해서야 소멸자가 터지며 조인 대기가 걸립니다. 그 전에 특정 시점에서 기다리고 싶다면 수동으로 `join()`을 해줘도 전혀 문제가 없습니다. (오히려 명시적이어서 좋습니다.)
2. **타임아웃과 강제 종료**: 
    실제 서버가 셧다운 될 때는 다음 순서를 따릅니다.
    * 운영자가 서버 셧다운 버튼을 누릅니다.
    * 메인 스레드가 100개의 워커 스레드에게 `request_stop()` 깃발을 흔듭니다.
    * 메인 스레드는 `join()`을 하기 전에, C++의 비동기 타임아웃 대기 기능(`future::wait_for` 등)을 이용해 **"딱 10초만 기다려준다"** 라고 초읽기를 시작합니다.
    * 10초 내에 스레드들이 DB 캐시를 저장하고 자기들 스스로 탈출하면 최고로 우아한 종료(Graceful Shutdown)가 완성됩니다.
    * 🚨 **만약 10초가 지났는데도 스레드가 반응이 없으면?**: 이 스레드는 데드락에 빠졌거나 무한 루프 버그에 걸린 '좀비 스레드'입니다. 이 경우 남은 상태 값을 로그로 뽑아내고, 스레드 단위가 아닌 **프로세스(서버) 전체를 OS 레벨에서 강제로 Kill(Abort)** 시켜버립니다. (스레드만 목 졸라 죽이는 것은 메모리 오염의 재앙을 낳으므로 C++은 이를 제공하지 않습니다.)

이처럼 스레드의 목숨줄을 밖에서 강제로 끊어버리는 것이 아니라, 깃발을 흔들어 스레드가 자기 할 일을 끝마치고 **스스로 목숨을 끊게(Return) 유도**하는 것이 가장 현대적이고 파워풀한 멀티스레드 설계 철학입니다.

## 🧠 심화 Q&A: `std::jthread`와 비동기 설계의 끝판왕

#### Q1. `std::stop_token`은 반드시 함수의 **첫 번째** 매개변수여야 하나요?
**예, C++20의 규칙입니다.** `std::jthread`는 내부적으로 템플릿 마법을 부려, 우리가 넘긴 함수의 첫 번째 매개변수 자리에 `std::stop_token` 타입이 기다리고 있으면 자기가 만든 깃발(수신기)을 **가장 첫 번째 인자로 몰래 끼워 넣어서** 함수를 실행시켜 줍니다.

#### Q2. `stoken.stop_possible()` 이 `false`가 나오는 실제 코드가 있나요?
이 함수는 **"내 깃발과 연결된 리모컨(`stop_source`)이 이 세상에 아직 살아 있는가?"** (취소될 가능성이 1%라도 있는가?) 를 묻는 함수입니다. 아래 2가지 상황에서 `false`가 나옵니다.
1. 메인 코드가 깡통 수신기만 띡 만들어서 넘겼을 때: `std::stop_token fake_token;` 
2. 스레드는 아직 살아서 돌고 있는데, 밖에서 나를 호출했던 리모컨(`stop_source` 객체나 `jthread` 객체)이 먼저 파괴되어 버렸을 때. (리모컨이 박살났으니 신호가 올 확률 0%)
*   **사용처**: 만약 `false`가 나온다면 "아 평생 멈출 일 없네" 하고 확신한 뒤, 루프 안에서 매번 검사하는 `stoken.stop_requested()` 로직 자체를 건너뛰어 **CPU 성능을 극한으로 최적화**할 때 쓰입니다.

#### Q3. 여러 스레드를 모두 한 번에 기다리거나 대기하지 않고, 완료되면 알아서 처리(비동기 콜백)하게 할 수는 없나요?
물론 가능합니다! 현업의 핵심 기술입니다.
1. **단일 함수로 대기 안됨**: Java처럼 여러 스레드를 파견하고 배열로 한 번에 묶어서 대기하는 단일 마법 함수는 C++ 표준에 없습니다. (일일이 반복문을 돌며 `.join()`을 하거나, C++20의 `std::barrier` / `std::latch` 같은 동기화 결승선 객체를 써야 합니다.)
2. **콜백(Callback)과 `std::future`**: 스레드에게 "다 끝나면 이 함수 실행해(Callback)" 라고 람다를 쥐여주는 비동기-이벤트 주도 방식이나, 메인 스레드가 진동벨(`std::future`)만 받아놓고 딴짓을 하다가 나중에 벨이 울릴 때 결과값만 쏙 빼먹는(`std::async`) 설계가 모던 비동기 프로그래밍의 표준입니다.

---

문서를 보시면서 `jthread`의 Auto-Join 개념과 `stop_token`을 사용한 우아한 종료 방식(Cooperative Cancellation)에 대해 궁금한 점이 있으신가요? 완벽히 이해하셨다면 바로 깃발을 흔들어서 백그라운드 스레드를 멈추는 최종 실습으로 들어가겠습니다!

### ⏱️ [Advanced] `50ms` 는 함수다? (chrono_literals)

`50ms` 같은 표현을 쓰려면 반드시 `using namespace std::chrono_literals;` 가 필요합니다.

**원리: 사용자 정의 리터럴 (User-Defined Literal)**
`50ms` 는 컴파일러에 의해 `operator""ms(50)` 이라는 함수 호출로 변환됩니다. 이 함수는 `std::chrono::milliseconds` 객체를 반환합니다.

**왜 네임스페이스가 필요한가요?**
- `ms` 같은 짧은 이름은 다른 라이브러리와 충돌할 위험이 큽니다.
- 따라서 C++ 표준은 이를 특정 서랍(`std::chrono_literals`)에 숨겨두고, 명시적으로 꺼내 쓰게끔 설계했습니다.
