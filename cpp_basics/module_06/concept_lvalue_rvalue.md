# L-value 와 R-value (그리고 `const` 참조자의 마법)

C++을 깊게 이해하기 위해, 그리고 나중에 배울 모던 C++의 핵심 기술(이동 시맨틱, `std::move`)을 마스터하기 위해 반드시 넘어야 할 산이 바로 **메모리의 방(주소) 관점**에서 값을 바라보는 이 개념입니다.

## 1. L-value (Locator Value) : "주소가 있는 진짜 방"

과거에는 대입 연산자(`=`)의 왼쪽(Left)에 올 수 있어서 L-value라고 불렀으나, 현대에는 **"Locator Value (주소로 가리킬 수 있는 뚜렷한 실체가 있는 값)"** 이라는 해석이 더 정확합니다.

*   `int myHp = 100;` 
*   `MagicType myType = MagicType::Fire;`
*   여기서 `myHp`와 `myType`은 명백한 L-value입니다. 시스템(RAM) 어딘가에 이들을 위한 고유한 방(주소, 예: `0x100`)이 할당되어 있고, 우리는 원할 때 언제든 `&myHp`처럼 앰퍼샌드를 붙여 그 주소를 알아낼 수 있습니다.
*   **특징:** 생명 주기(Lifetime)가 블록(`{ }`)을 벗어날 때까지 길게 지속됩니다.

## 2. R-value (Read-only Value / 임시 값) : "방이 없는 허공의 데이터"

대입 연산자(`=`)의 오른쪽(Right)에만 올 수 있는 수식을 의미하며, 최근에는 **"Read-only Value (단순히 읽기만 가능한 이름 없는 임시 값)"**으로 불립니다.

*   `100` (숫자 리터럴 그 자체)
*   `MagicType::Fire` (열거형 값 그 자체)
*   `myHp + 50` (덧셈을 하고 난 직후의 계산 결과물)
*   이 녀석들은 메모리 상의 '방(주소)'을 가지지 않습니다. CPU 내부의 작업 공간(Register)에 찰나의 순간 электри신호로 존재했다가, 세미콜론(`;`)을 만나 문장이 끝나는 순간 연기처럼 뿅 하고 증발해버립니다.
*   **특징:** 방 번호가 없기 때문에 **`&(MagicType::Fire)` 처럼 주소를 빼오려고 시도하면 컴파일러가 강력하게 거부(에러)**합니다.

---

## 3. 참조자(`&`)의 룰과 충돌: "가리킬 방이 안 보이는데?"

함수에서 큰 데이터를 넘길 때 복사(Copy) 비용을 피하려고 보통 일반 참조자(L-value Reference)를 사용합니다.

```cpp
void CalculateTotalDamage(MagicType& type); // 참조자(방의 별명)로 받겠다!
```

이 함수에 L-value 였던 `myType` 변수를 넘기는 건 아주 잘 작동합니다. (원본 방의 주소를 아니까요). 
하지만 **허공의 기호(R-value)를 던지면 대형 사고가 납니다.**

```cpp
CalculateTotalDamage(MagicType::Ice); // ❌ 컴파일 에러 발생!
```
*   **에러 원인:** `MagicType::Ice`는 방(메모리)이 없는 R-value 입니다. `CalculateTotalDamage` 함수는 "어떤 방의 주소(원본)를 가져와서 내가 임의로 고쳐 쓸 수 있는 상태(`&`)"를 요구하는데, 애초에 고쳐 쓸 수 있는 **방 자체가 존재하지 않으니 컴파일러가 분노**하며 에러를 내뱉습니다.

---

## 4. `const &` (상수 참조자의 마법) : 생명 연장(Lifetime Extension)

방 없는 임시 데이터(R-value)를 참조자처럼 "복사 없이" 넘기고 싶을 때는 어떻게 해야 할까요?
여기서 C++ 컴파일러가 아주 특별한 혜택(마법)을 제공합니다. 바로 **`const`를 붙이는 것**입니다.

```cpp
// "나는 넘겨받은 원본 방을 절대 수정(쓰기) 안 하고 오직 읽기(const)만 할게!"
void CalculateTotalDamageConst(const MagicType& type); 
```

```cpp
CalculateTotalDamageConst(MagicType::Ice); // 🟢 통과 (에러 안 남!)
```

*   **배후의 마법:** 컴파일러가 코드를 번역하다가, R-value(`MagicType::Ice`)를 `const &`로 넘긴다는 사실을 발견합니다. 
*   **컴파일러의 독백:** *"아하! 이 녀석이 수정 안 하겠다고 약속(`const`)했구나. 근데 저 값은 방이 없어서 주소를 넘길 수가 없잖아? 할 수 없지. 내가 몰래 스택(Stack) 메모리의 은밀한 어딘가에 **`Ice`를 담을 수 있는 이름 없는 임시 방을 파주고**, 그 방의 주소를 참조자가 가리키게 해주지. 그리고 이 함수가 끝날 때까지 방을 부수지 않고 (Lifetime Extension, 생명 연장) 유지시켜 줄게!"*

이 방식 덕분에 거대한 임시 객체들(`string("Hello")` 등)을 넘길 때 일일이 값 복사를 일으키지 않고도 함수 안으로 쏙 넣어줄 수 있습니다.

### 💡 실무 관점에서의 요약 (핵심 팁)

1.  수정이 불가능한 허공의 값(R-value)을 넘기고 싶다면, 함수 인자는 반드시 **`const` 참조자**로 만들어야 합니다.
2.  하지만 4바이트짜리 `int`나 `enum` 같은 작은 기본 타입들은 굳이 컴파일러가 보이지 않는 방을 몰래 파주게 만드는 노력조차 아까울 정도로 크기가 작습니다.
3.  따라서, **"작은 데이터 타입은 그냥 Call by Value(값 복사)로 `(MagicType type)` 남기는 것이 무조건 가장 빠르고 오버헤드가 적은 정석(Best Practice)"**입니다. 반대로 덩치가 큰 `struct`, `class`, `std::string`은 반드시 `const &`로 넘기세요!
