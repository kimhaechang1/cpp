# [심화] 3/5/0 규칙 (Rule of 3/5/0)

우리가 지금까지 C++의 메모리와 클래스를 다루면서 가장 고통받았던 순간을 떠올려 봅시다.
"생성자에서 `new`를 했으면 소멸자에서 `delete`를 해야 하고... 어? 다른 객체로 복사할 땐 복사 생성자도 똑같이 만들어줘야 하고... 대입할 땐 대입 연산자도 오버로딩해야 하고..."

한 번 힙(Heap) 메모리에 손을 대기 시작하면, 처리해야 할 뒷수습이 폭포수처럼 쏟아졌습니다. 이 복잡한 관리 규칙을 명문화한 것이 바로 **C++의 Rule of Three (3의 규칙)**입니다.

---

## 1. Rule of Three (3의 규칙) : "하나를 건드렸으면 셋 다 책임져라"

C++98 시절부터 내려져 온 아주 무서운 철칙입니다.
> **"만약 당신의 클래스에서 [소멸자], [복사 생성자], [복사 대입 연산자] 중 단 하나라도 직접 구현할 필요가 생겼다면, 당신은 반드시 이 3가지를 모두 직접 구현해야만 버그(메모리 누수, 크래시)를 막을 수 있다."**

**왜 그럴까요?**
우리가 이 셋 중 하나를 건드렸다는 것은 **"이 클래스가 단순한 값(숫자)이 아니라, 포인터(`new`) 같은 위험한 자원을 수동으로 직접 관리하고 있다"**는 명백한 증거이기 때문입니다.

1. **소멸자**: 내가 직접 `delete`를 해야 해서 만들었다면?
2. **복사 생성자**: 디폴트 얕은 복사(`=`)를 쓰면 포인터 주소만 복사돼서 이중 해제(Double Free) 크래시가 나니까, 내가 직접 깊은 복사(Deep Copy)를 만들어야 합니다.
3. **복사 대입 연산자**: 기존에 들고 있던 메모리를 지우고 새 메모리를 할당해서 덮어씌워야 하니까 꼭 필요합니다.

---

## 2. Rule of Five (5의 규칙) : "이제 5개다"

이후 모던 C++(C++11)로 넘어오면서 **이동 시맨틱(Move Semantics)**이라는 기술이 추가되었습니다. (효율성을 위해 복사하지 않고 소유권만 뺏어오는 마법의 기술인데, 나중에 자세히 배웁니다!)
기술이 추가되면서 챙겨야 할 함수가 2개 더 늘어났습니다.

> **"수동으로 자원을 관리할 거라면, [소멸자], [복사 생성자], [복사 대입 연산자], [이동 생성자], [이동 대입 연산자] 이 5가지를 전부 다 당신이 완벽하게 코딩해야 합니다!"**

숨이 턱 막히지 않으시나요? 고작 슬라임 하나를 관리하기 위해 똑같은 보일러플레이트 코드 5개를 벅벅 작성해야 합니다. 이쯤 되면 "아 기껏 만들어놓은 거 그냥 C# 이나 Java처럼 알아서 쓰레기 청소(Garbage Collection) 해주면 안 돼?" 라는 불만이 C++ 개발자들 사이에서 폭발하기 시작했습니다.

---

## 3. ✨ 구원자, Rule of Zero (0의 규칙) : "아무것도 하지 마라!"

C++ 창시자(Bjarne Stroustrup)를 비롯한 위원회는 "인간은 이 5개를 완벽하게 관리할 수 없다"고 공식적으로 패배를 선언(?)하고, C++11부터 미친 패치를 단행했습니다.

> **"프로그래머여, 이제 저 5개의 함수를 단 한 개도(Zero) 직접 적지 마라. 대신 포인터를 쓰지 말고, 알아서 메모리를 관리해 주는 똑똑한 부품들을 써라!"**

### 💡 Rule of Zero의 핵심 실천법
클래스의 멤버 변수로 생짜 포인터(`int*`, `Slime*`) 나 배열(`int array[10]`)을 **절대 사용하지 않습니다.**
대신 이미 누군가가 저 5개의 규칙을 완벽하게 다 짜놓아서 절대 메모리가 안 새는 **"표준 라이브러리(STL)"** 부품으로 전부 교체합니다!

1. 문자열 포인터(`char*`) -> **`std::string`**
2. 동적 배열 포인터(`int*`) -> **`std::vector`**
3. 객체 주소 포인터(`Slime*`) -> **`std::unique_ptr` (스마트 포인터)**

```cpp
#include <iostream>
#include <string>
#include <vector>
#include <memory> // 스마트 포인터를 위해 필요

class Slime {
};

// [과거 생지옥 클래스: Rule of 5 억지로 써야 함]
class OldMonster {
private:
    char* name;
    Slime* mySlime;
public:
    OldMonster() { /* new 할당 */ }
    ~OldMonster() { /* delete 해제 */ }
    OldMonster(const OldMonster& m) { /* 깊은 복사 구현 */ }
    // ... 대입 연산자, 이동 생성자 등등 지옥도 ...
};

// =======================================================
// [✨ 현대 C++ 우아한 클래스: Rule of Zero]
class ModernMonster {
private:
    std::string name;                          // char* 포기!
    std::vector<int> inventory;                // int* 포기!
    std::unique_ptr<Slime> bossSlime;          // 생짜 Slime* 포기!
    
public:
    // 생성자에서 초기화만 해주면 끝!
    ModernMonster(std::string n) : name(n) {
        // 스마트 포인터 초기화 (뒤에서 배움!)
        bossSlime = std::make_unique<Slime>(); 
    }

    // 소멸자? 안 씁니다. (컴파일러가 알아서 지워줌)
    // 복사 생성자? 안 씁니다. (컴파일러가 알아서 복사해 줌)
    
    // 이 클래스는 메모리 누수 발생 확률 0%의 완벽한 안전 지대가 되었습니다!
};
```

### 🎯 결론
**"최고의 메모리 관리는, 내가 직접 메모리를 관리하지 않는 것이다."**

이 철학이 바로 모던 C++의 심장이자 알파와 오메가입니다. 
우리가 앞에서 생고생을 하며 `new`와 `delete`, 복사 생성자를 배웠던 이유는, "이게 왜 필요한가?"를 깨닫기 위해서, 그리고 아주 드물게 직접 엔진의 바닥 자원을 설계해야 하는 순간(Socket, Thread 등)을 대비하기 위함이었습니다.

하지만 일반적인 상황에서는 무조건 **Rule of Zero**를 따라야 합니다.
이제 비로소 우리는 메모리 관리의 공포에서 해방되어, 다음 모듈부터는 본격적으로 저 똑똑한 부품들(STL 자료구조와 스마트 포인터)을 조립하는 즐거운 코딩을 시작할 수 있게 되었습니다!

## 4. [부록] 억지로 막기: `= delete` (C++11)

가끔은 포인터를 쓰지는 않지만, **논리적으로 복사되면 절대 안 되는 객체**(예: 유일한 전투 매니저, 네트워크 연결 소켓, 식별자(ID)를 가진 단일 발사체)가 있습니다. 
이럴 때는 컴파일러가 몰래 '기본 복사 생성자'를 만들어주는 호의조차 재앙이 됩니다.

과거(C++98)에는 이를 막기 위해 복사 생성자를 `private:`에 숨기고 구현을 아예 안 하는 꼼수를 썼습니다. 하지만 이제 모던 C++에서는 **명시적으로 생성을 금지하는 단어**가 추가되었습니다.

```cpp
class UniqueBoss {
public:
    UniqueBoss() {} // 기본 생성자는 허용

    // "파괴자(Compiler)여, 이 클래스의 복사 기능은 영원히 삭제하라!"
    UniqueBoss(const UniqueBoss&) = delete;            // 복사 생성자 삭제
    UniqueBoss& operator=(const UniqueBoss&) = delete; // 대입 연산자 삭제
};
```

이렇게 명시해두면, 누군가 실수로 `UniqueBoss b2 = b1;` 을 치는 순간 컴파일러가 단호하게 **"이 함수는 지워졌습니다(deleted)"** 라며 컴파일 에러를 띄워줍니다. 의도가 100% 명확한, 완벽한 방어막입니다!

---

### ✨ [Advanced] `= default` 의 정체 (M11 심화)

`= default` 는 컴파일러에게 **"이 함수의 기본 구현을 네가 알아서 만들어줘"** 라고 지시하는 문법입니다.

```cpp
virtual ~Monster() = default;  // 컴파일러에게 기본 소멸자 생성을 위임
virtual ~Monster() {}          // 직접 빈 본문을 작성 (기능적으로 동일)
```

**왜 `{}` 대신 `= default` 를 쓰나요?**

1. **의도 표현**: "나는 기본 동작을 원한다"는 **의도를 명확히 표현**합니다. `{}` 는 실수로 비워둔 건지 알기 어렵습니다.
2. **최적화**: 컴파일러가 "사소한(trivial)" 함수로 판단하여 더 나은 최적화를 적용할 수 있습니다.
3. **가독성**: `= delete`(금지)와 대조되어 클래스의 인터페이스 정책을 한눈에 보여줍니다.
