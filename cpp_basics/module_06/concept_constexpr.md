# `constexpr`: 컴파일 타임 연산 (Zero-Overhead Abstraction)

## 🎮 실무 프로젝트 상황: 런타임 계산의 병목
게임이 시작될 때 '무기 10,000종'의 밸런스 테이블을 구축하거나, 특정 수학적 연산(예: 팩토리얼 계산, 진법 변환, 복잡한 상수 도출)을 해야 한다고 가정해봅시다. 
이러한 연결을 게임이 켜진 후(Runtime)에 수행하면, 유저 입장에서는 **로딩 화면이 길어지거나 프레임 드랍(Frame Drop)**을 경험하게 됩니다. 

*과연 "결과가 이미 정해져 있는 계산"을 굳이 게임 실행 중에 해야 할까요?*

---

## 💡 이론: `constexpr`란 무엇인가?

`constexpr`(Constant Expression, 상수 상수식)는 C++11에서 도입되어 모던 C++의 최적화 핵심으로 자리 잡은 키워드입니다.
이 키워드는 컴파일러에게 다음과 같이 강력하게 지시합니다.

> **"이 변수의 값(또는 함수의 결과)은 무조건 컴파일 시간(Compile Time)에 미리 계산해서, 프로그램(실행 파일) 안에 '계산이 끝난 상숫값'으로 하드코딩해버려라!"**

### 1. `const` vs `constexpr`의 근본적 차이점

*   **`const` (Read-Only)**
    *   "이 변수는 초기화된 이후에는 값을 변경할 수 없다"는 의미입니다.
    *   컴파일 타임에 값이 정해질 수도 있지만, **런타임에 값이 정해지는 것도 가능**합니다. (예: 사용자 입력을 받아 `const` 변수에 초기화)
*   **`constexpr` (Compile-Time Evaluation)**
    *   Read-Only는 기본 탑재이며, **반드시 컴파일 타임에 값을 결정할 수 있어야 한다**는 훨씬 엄격한 제약이 걸립니다.
    *   성능 관점에서 게임 실행 시점(Runtime)에는 연산 시간 단축 효과(연산 비용 0, Zero-Overhead)를 가져옵니다.

### 2. 컴파일러의 마법: 런타임 비용 **Zero**

```cpp
#include <iostream>

// 일반 함수 (실행 중에 연산 발생)
int GetMaxHpRuntime(int level) {
    return level * 100 + 50;
}

// constexpr 함수 (조건 충족 시 컴파일 타임에 연산 종료)
constexpr int GetMaxHpCompileTime(int level) {
    return level * 100 + 50;
}

int main() {
    // 1. 일반 함수 호출 -> 프로그램이 실행되면서 10 * 100 + 50을 계산합니다.
    int hp1 = GetMaxHpRuntime(10); 

    // 2. constexpr 함수 호출 -> 10은 상수이므로, 컴파일러가 미리 1050으로 계산해버립니다.
    // 어셈블리(어셈블리어) 레벨에서 보면 이 코드는 사실상 `constexpr int hp2 = 1050;` 과 동일해집니다.
    constexpr int hp2 = GetMaxHpCompileTime(10); 
    
    std::cout << "HP1: " << hp1 << ", HP2: " << hp2 << "\n";
    return 0;
}
```

#### 🔬 하드웨어/메모리 관점 검증 (어셈블리 최적화)
`constexpr`로 작성된 코드를 어셈블리어로 변환하여 뜯어보면, `GetMaxHpCompileTime(10)` 이라는 함수를 호출(`call`)하는 과정(스택 프레임 생성, 매개변수 전달 등)이 **아예 사라져 있습니다.**
대신 레지스터에 `1050`이라는 값 자체를 바로 집어넣는(`mov`) 단 한 줄의 명령어로 치환됩니다. 이것이 바로 모던 C++이 추구하는 **Zero-Overhead Abstraction**입니다.

---

## ⚠️ `constexpr` 함수의 엄격한 제약 조건 (C++14/20 기준)

`constexpr` 함수가 컴파일 타임에 계산되려면 컴파일러가 모든 걸 미리 알 수 있어야 하므로, 함수 내부에 제약이 존재합니다. (C++ 버전이 오르며 제약이 많이 풀리고 있습니다)

1.  **런타임에만 알 수 있는 값은 불가능**: 난수 생성 장치(랜덤), 사용자 입력(`cin`) 등은 컴파일 타임에 알 수 없으므로 `constexpr` 연산에 포함될 수 없습니다.
2.  **동적 할당 제약**: `new`/`delete` 같은 동적 할당은 런타임 운영체제의 도움이 필요하므로 전통적으로 금지되었습니다. (※ C++20부터는 제약 조건부 허용되는 트렌드로 발전 중이지만, 기본적으로는 피하는 것이 좋습니다.)

## 🚀 심화: "Java의 상수 접합과 비슷하네요. 단순 계산 말고 더 있나요?"

네! 회창님이 정확히 짚어주셨습니다. Java의 컴파일러 최적화(상수 폴딩, String 리터럴 결합 등)와 시작점은 비슷합니다. 하지만 **C++의 `constexpr`은 그 한계를 벗어나 사실상 '컴파일 타임에 도는 또 다른 미니 C++ 프로그램' 수준으로 진화했습니다.**

단순한 숫자 더하기를 넘어서, C++에서 `constexpr`이 막강한 이유는 다음과 같습니다.

### 1. C++ 문법이 "반드시 컴파일 타임 상수"를 요구하는 곳에 자유롭게 사용 가능
C++에서는 반드시 컴파일할 때 크기가 정해져 있어야 하는 문법들이 있습니다.
*   **고정 배열의 크기 지정** (`int arr[size];`)
*   **템플릿(Template) 파라미터** (추후 배울 내용)

일반적인 함수(`GetMaxInventorySize()`)를 배열의 크기로는 절대 쓸 수 없지만, `constexpr` 함수라면 가능합니다!

```cpp
constexpr int GetMaxInventorySize(int base, int bonus) {
    return base + bonus;
}

// 런타임 함수였다면 에러가 나지만, constexpr이므로 컴파일러가 미리 20으로 계산하여 배열을 만듭니다.
int playerInventory[GetMaxInventorySize(10, 10)]; 
```

### 2. 객체(Object) 자체를 컴파일 타임에 굽기 (`constexpr` 생성자)
이 부분이 Module 06 클래스 단원과 연결되는 핵심입니다. 숫자 한두 개가 아니라, 아예 **객체 자체를 게임이 시작되기도 전에 램(RAM) 형태 그대로 실행 파일(.exe) 안에 구워(Bake)** 버릴 수 있습니다!

```cpp
class Vector3 {
public:
    double x, y, z;

    // 생성자에도 constexpr을 붙일 수 있습니다!
    constexpr Vector3(double x, double y, double z) : x(x), y(y), z(z) {}
    
    constexpr double GetMagnitudeSquare() const {
        return (x*x) + (y*y) + (z*z);
    }
};

// 프로그램이 실행될 때 객체를 생성하는게 아님. 
// 이미 (1.0, 2.0, 3.0)이 채워진 상태로 .exe 안에 저장됨!
constexpr Vector3 FORWARD_VECTOR(1.0, 2.0, 3.0); 

// 이 함수 호출조차 컴파일할 때 끝나서, 어셈블리에는 '14.0'이라는 결과만 박힙니다.
constexpr double mag = FORWARD_VECTOR.GetMagnitudeSquare(); 
```
이처럼 상수로 고정되는 몬스터의 초기 스탯 테이블, 파이프라인 설정 데이터, 기본 방향 벡터 등을 `constexpr` 객체로 만들면 로딩 시간이 비약적으로 줄어듭니다.

### 3. 컴파일 타임 분기점: `if constexpr` (C++17)
코드가 실행될 때 `if`문을 타는 것이 아니라, **컴파일할 때 조건을 판단하여 안 쓰는 코드 블록을 통째로 날려버리는(Discard) 기능**입니다. 덕분에 조건문의 런타임 오버헤드가 단 1도 발생하지 않습니다. (추후 템플릿 최적화에서 게임 체인저로 활약합니다).

**요약하자면:** 
Java의 상수 계산은 컴파일러가 '알아서 해주는 최적화'의 성격이 짙다면, C++의 `constexpr`은 **"프로그래머가 컴파일러에게 명시적으로 명령을 내려 아예 컴파일 시간에 코드를 실행(Evaluate)시켜버리는 강력한 도구"**입니다. 심지어 C++20부터는 컴파일 타임에 `std::vector`를 만들어서 `push_back`으로 데이터를 채워 넣고 그걸 결과 배열로 뱉어내는 것까지 가능해졌습니다!
