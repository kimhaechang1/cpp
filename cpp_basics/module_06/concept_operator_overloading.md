# 연산자 오버로딩과 Rule of Three (대입 연산자)

## 🚨 상황 극장: "복사 생성자 짰으니까 이제 안전하겠지?"

지난 시간에 우리는 **복사 생성자(Copy Constructor)** 를 만들어 `Double Free(이중 해제)` 문제를 완벽하게 막아냈습니다.
그래서 회창님은 안심하고 게임 로그인을 구현하다가 아래와 같은 코드를 작성하게 됩니다.

```cpp
// 1. 플레이어 A 생성 (초기화)
Inventory playerA;
playerA.AddItem("집행검");

// 2. 플레이어 B 생성 (미리 만들어 둠)
Inventory playerB;
playerB.AddItem("나무 몽둥이");

// 3. A가 B에게 모든 템을 넘겨줌 (대입)
playerB = playerA; 
```

**질문:** 위 코드는 안전할까요? 
정답은 **"아니오, 또 펑 터집니다!"** 입니다. 💥 

분명 깊은 복사(Deep Copy)를 만들었는데 왜 터질까요? 
그 이유는 **`초기화(Initialization)`** 와 **`대입(Assignment)`** 이 완전히 다른 동작이기 때문입니다.

*   `Inventory B = A;` ➡️ **초기화**. 태어날 때 A를 똑같이 따라 함. (**복사 생성자** 호출)
*   `B = A;` ➡️ **대입**. 이미 태어나서 살고 있는 B가, 자기 걸 다 버리고 A의 상태로 갈아탐. (**대입 연산자** 호출)

우리는 '복사 생성자'만 만들었지, '대입 연산자'는 만들지 않았습니다. 그래서 컴파일러가 또 멋대로 **"얕은 복사(Shallow Copy)"** 대입을 해버린 것입니다.

---

## 🛠️ 연산자 오버로딩 (Operator Overloading) 이란?

C++의 가장 강력한 특징 중 하나입니다.
`+`, `-`, `*`, `/`, `=`, `==` 같은 기호들을 단순한 숫자가 아니라, **우리가 만든 객체(클래스/구조체)** 에도 사용할 수 있게 의미를 재정의(Overloading)하는 기능입니다.

사실 `playerB = playerA;` 라는 코드는 컴파일러 눈에는 다음과 같은 **함수 호출**로 보입니다.

```cpp
playerB.operator=(playerA); // operator= 라는 이름의 함수를 호출!
```

## ✨ 대입 연산자 (`operator=`) 직접 구현하기

메모리 누수와 이중 해제를 막기 위해, 컴파일러가 만들어주는 얕은 복사 대입 연산자를 버리고 우리가 직접 딥 카피(Deep Copy)를 하는 대입 연산자를 만들어야 합니다.

**형태:**
```cpp
class Inventory {
    int* items;
    // ... 생략 ...

public:
    // 대입 연산자 오버로딩
    Inventory& operator=(const Inventory& other) {
        // 여기에 깊은 복사 로직을 작성합니다.
    }
};
```

### 🧠 대입 연산자 작성 시 주의할 3가지 (매우 중요)

복사 생성자와 비슷해 보이지만, **이미 존재하던 객체**에 값을 덮어씌워야 하므로 훨씬 까다롭습니다.

#### 1. 나 자신을 대입하는지 확인 (Self-Assignment Check)
만약 누군가 실수로 `playerA = playerA;` 라고 쓰면 어떻게 될까요? 
덮어씌우기 전에 원래 내 데이터를 지워버리기 때문에 큰일이 납니다. 그래서 내 주소(`this`)와 들어온 놈의 주소(`&other`)가 같은지 먼저 검사해야 합니다.

#### 2. 기존 메모리 해제 (Clean up old resources)
초기화(복사 생성자) 때는 내가 빈털터리였지만, 대입될 때는 내가 이미 Heap 메모리를 가지고 있을 수 있습니다. (위 예시의 "나무 몽둥이").
이걸 지우지 않고 새 메모리를 덮어씌우면 "나무 몽둥이"는 평생 메모리 미아가 됩니다 (**Memory Leak!**). 반드시 먼저 `delete` 해야 합니다.

#### 3. 깊은 복사 수행 & 자기 자신 반환
새 메모리를 할당하고 내용을 복사한 뒤, **자기 자신을 나타내는 참조(`*this`에 대한 참조)**를 반환합니다.
왜 반환할까요? C++에서는 `A = B = C;` 처럼 연쇄적인 대입이 가능해야 하기 때문입니다.

### 📝 완성된 대입 연산자

```cpp
Inventory& operator=(const Inventory& other) {
    // 1. 나 자신 대입 방지
    if (this == &other) {
        return *this; 
    }

    // 2. 기존에 내가 들고 있던 메모리 해제 (메모리 누수 방지)
    if (this->items != nullptr) {
        delete[] this->items;
    }

    // 3. 깊은 복사 (크기 베끼고, 새 메모리 파고, 내용물 복사)
    this->capacity = other.capacity;
    this->items = new int[this->capacity];
    for (int i = 0; i < this->capacity; i++) {
        this->items[i] = other.items[i];
    }

    // 4. 나 자신(참조) 반환 (A = B = C 지원)
    return *this;
}
```

## 🏛️ Rule of Three (3의 법칙)

C++ 메모리 관리의 가장 유명한 황금률입니다. 현업 면접에서도 단골로 나오는 주제입니다.

**어떤 클래스가 힙(Heap) 메모리를 사용해서 포인터를 가지고 있다면, 다음 3가지는 묻지도 따지지도 말고 "무조건 세트로" 함께 만들어야 한다.**

1.  **소멸자 (Destructor):** 객체가 죽을 때 메모리 반환
2.  **복사 생성자 (Copy Constructor):** 객체를 기반으로 새로 태어날 때 깊은 복사
3.  **대입 연산자 (Copy Assignment Operator):** 이미 살아있는 객체에 다른 객체를 덮어씌울 때 깊은 복사

단 하나라도 빼먹으면 메모리 누수(Memory Leak)이거나 이중 해제(Double Free)로 인한 크래시가 반드시 발생합니다.

---

> **💡 Knowledge Connectivity**: 대입 연산자에서 `if (this == &other)`를 체크하고, 기존 메모리를 `delete`하고, 다시 `new` 하는 과정은 매우 복잡하고 버그가 생기기 쉽습니다. (예: `new`에서 예외 발생 시 기존 데이터 상실)
> 현대적 C++ 개발자들은 이런 원시적인 포인터 조작을 직접 하지 않고, **`std::unique_ptr`** 등을 멤버로 사용하여 컴파일러에게 이 모든 책임을 떠넘기는 **'0의 규칙(Rule of Zero)'**을 사용합니다.

## 🙋‍♂️ Q&A (자주 묻는 질문)

### Q1. `this`는 자기 자신을 가리키는 포인터인데, `*this`는 무엇인가요?
*   `this`: 현재 객체의 **메모리 주소**(포인터)입니다. (예: `0x7ffee4b5c9a0`)
*   `*this`: 그 주소에 있는 **진짜 객체 즉, "나 자신"의 알맹이(값)** 를 의미합니다. (역참조)
    *   우리가 대입 연산자에서 `Inventory& operator=()`라고 선언하여 **참조(&)** 를 반환하기로 했으므로, 포인터인 `this`를 그대로 반환하면 타입이 안 맞습니다.
    *   대신 `*this`를 반환하면, 나의 알맹이를 참조(Reference)로 반환하는 것이 되어 깔끔하게 문법이 맞아떨어집니다.

### Q2. `A = B = C;`를 하면 어떤 순서로 연산자 오버로딩이 작동하나요?
대입 연산자(`=`)는 기본적으로 **오른쪽에서 왼쪽(Right-to-Left)** 으로 결합(수행)됩니다.
따라서 `A = B = C;`는 C++ 컴파일러 눈에 다음과 같이 보입니다.
1.  먼저 **`B = C`** 가 실행됩니다. `B.operator=(C)`가 호출되어 C의 내용이 B에 깊은 복사됩니다.
2.  이때 `B.operator=(C)`는 자기 자신인 **B의 참조(`*this`)를 반환**합니다.
3.  그 반환된 결과(B)를 가지고 이제 **`A = B`** 가 실행됩니다. `A.operator=(B)`가 호출되어 방금 복사된 B의 내용이 A에 다시 깊은 복사됩니다.

이처럼 **연쇄적인 호출**이 가능하도록 만들기 위해서 대입 연산자는 반드시 `*this` (자기 자신의 참조)를 반환해야 하는 것입니다!

### Q3. 왜 반환 타입을 `Inventory` (값 반환)로 하면 복제가 일어나나요? 그냥 자기 자신 인스턴스를 반환하면 안 되나요?
함수가 종료되면서 값을 반환할 때 작동하는 **C++의 본질적인 "값에 의한 전달(Pass by Value)" 방식** 때문입니다.
1.  함수(이 경우 대입 연산자)가 어떤 데이터를 뱉어내려(return) 할 때, 그 데이터를 잠시 담아둘 **임시 공간**이 메모리에 생깁니다.
2.  `return *this;`를 실행하지만, 반환 타입이 `Inventory` 라면, C++ 컴파일러는 내 진짜 알맹이(`*this`)를 **그 임시 공간으로 "복사"해서 던져줍니다.** (이때 원하지 않았던 **복사 생성자**가 강제로 호출됩니다!)
3.  왜 그대로 안 주고 굳이 복사해서 던질까요? 함수가 끝나면(return) 원래 함수 내부에서 썼던 변수들이 다 죽어버리는 게 C++ 세상의 이치니까, 컴파일러가 "안전하게 복사본을 전달해서 밖에서 쓰게 하자"라고 아주 보수적으로 설계되어 있기 때문입니다.
4.  반면, 반환 타입을 `Inventory&` (참조 반환)로 설정하면, C++은 임시 공간에 무거운 복사본을 만들지 않습니다. 대신 **"진짜 원본 객체가 원래 살고 있는 메모리 위치(참조)"** 만 살짝 건네주고 끝납니다.

따라서 힙 메모리 복사라는 엄청난 성능 낭비를 막고 원래 의도(진짜 B 자신을 A에게 넘겨줌)를 달성하기 위해 **무조건 `&`(참조)** 를 붙여서 반환하는 것입니다!

### Q4. 어차피 `this` 포인터도 함수 안에서만 쓰이는 매개변수 같은 거니까, `return *this;` 를 값으로 반환하면 과연 무엇이, 어떻게 복사된다는 말인가요?
정말 예리한 질문입니다! 여기서 "복사된다"는 것은 `this`라는 포인터 변수(주소표지판) 자체가 복사된다는 의미가 아닙니다.
`*this` 를 역참조해서 찾아낸 **거대한 객체의 알맹이 전체** 똑바로 복사된다는 뜻입니다.

1.  함수가 값(Inventory)을 반환하려 할 때, 컴파일러는 메모리 어딘가에 **`임시 Inventory 객체`** 를 하나 몰래 만듭니다.
2.  그리고 우리가 지난 시간에 피땀 흘려 만들었던 **복사 생성자(Copy Constructor)** 를 몰래 호출합니다!
    *   `Inventory 임시객체 = *this;` 처럼 동작하는 겁니다.
3.  이때 무슨 일이 벌어질까요?
    *   임시 객체 안의 힙 메모리 주소(`items`)를 위해 **새로운 `new int[]`가 실행**됩니다!
    *   원본(`*this`)에 있던 수백 개의 아이템 데이터가 `for`문을 돌면서 **그대로 임시 객체로 깊은 복사**됩니다.
4.  함수가 끝나고 밖으로 던져지는 건 원본 B 객체가 아니라, 방금 저 엄청난 고생을 해서 만든 **임시 객체**입니다.
5.  그리고 대입이 끝나면, 그 임시 객체는 할 일 다 했다며 **소멸자(Destructor)** 를 부르며 사라집니다.

즉, "this가 복사된다"가 아니라, **"this가 가리키는 원본 객체의 알맹이를 통째로 떠서(=깊은 복사를 한 번 더 해서) 밖으로 넘겨준다"**는 뜻입니다.
우리가 왜 기를 쓰고 반환 타입에 `&`(참조)를 붙여서 이 끔찍한 추가 작업을 막으려 하는지 이제 100% 이해되셨을 겁니다! 🚀
