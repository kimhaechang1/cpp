# 스마트 포인터 미리보기: `std::unique_ptr`

C++11에서 등장한 **스마트 포인터(Smart Pointer)**는 "포인터의 탈을 쓴 영리한 클래스(객체)"입니다.
가장 대표적이고 현업에서 90% 이상 사용되는 것이 바로 **`std::unique_ptr` (독점적 스마트 포인터)** 입니다. 

---

## 1. 생짜 포인터의 문제점 (복습)
```cpp
void PlayGame() {
    Slime* slime = new Slime(); // Heap에 생성
    
    if (BossAppeared) {
        return; // 🚨 치명적 실수! delete를 안 하고 함수가 끝나버림 (메모리 누수 발생)
    }
    
    delete slime; // 무사히 도달해야만 해제됨
}
```
위처럼 생짜 포인터(`Raw Pointer`)는 프로그래머가 모든 실행 경로(if, return, 에러 발생 등)에서 무조건 `delete`를 챙겨줘야 하는 끔찍한 단점이 있었습니다.

---

## 2. RAII의 결정체: `std::unique_ptr`

> **RAII (Resource Acquisition Is Initialization)란?**
> "자원의 획득은 초기화다"라는 뜻으로, C++에서 골머리를 앓게 만드는 메모리, 파일, 네트워크 소켓 등의 위험한 자원들을 **안전한 객체(지역 변수)로 한 번 감싸서**, 객체가 생겨날 때(생성자) 자원을 얻고 객체가 죽을 때(소멸자) 알아서 자원을 반납하게 만드는 C++ 최고의 핵심 설계 패턴입니다.

스마트 포인터는 우리가 방금 배운 **"지역 변수(Stack)는 블록 `{}`이 끝나면 즉시 소멸자가 불린다"**는 C++의 기본 원칙(RAII)을 극강으로 활용한 녀석입니다.

*   `std::unique_ptr` 자체는 **Stack(지역 변수) 영역**에 생성됩니다.
*   하지만 그 녀석의 "뱃속(내부 멤버 변수)"에 우리가 만든 **Heap(동적 할당) 메모리의 주소**를 품고 있습니다.
*   함수가 끝나서 Stack에 있던 `unique_ptr`이 수명을 다해 자기 소멸자를 호출할 때, **그 소멸자 내부에서 자기가 품고 있던 Heap 주소에 `delete`를 자동으로 쏴줍니다!**

```cpp
#include <memory> // 필수 헤더

void PlayGame() {
    // 1. make_unique를 통해 Heap에 Slime을 만들고, 포인터를 스마트 포인터에 담음
    std::unique_ptr<Slime> slime = std::make_unique<Slime>();
    
    if (BossAppeared) {
        // 걱정 마세요! 함수를 빠져나가는(return) 순간 지역 변수 'slime'이 파괴되면서,
        // 내부적으로 자기가 품고 있던 Heap 슬라임을 자동으로 delete 해버립니다!
        return; 
    }
    
    // delete slime; <-- 다 안 지워도 됩니다. 인간은 이제 자유입니다.
} // 함수 끝 -> 여기서도 자동 delete 됨!
```

## 3. 왜 "Unique(독점적)" 포인터인가?

메모리가 두 번 해제되는(Double Free) 끔찍한 버그를 원천 차단하기 위해, `unique_ptr`은 **"복사(Copy)를 문법적으로 절대 금지"** 시켜버렸습니다.

```cpp
std::unique_ptr<Slime> slime1 = std::make_unique<Slime>();

// 🚨 컴파일 에러! "deleted function" (독점적 소유권이라 남에게 복사해 줄 수 없음)
std::unique_ptr<Slime> slime2 = slime1; 
```

오직 단 한 명의 주인만 그 Heap 메모리를 가리킬 수 있도록 보장합니다. 주인이 죽으면 메모리도 확실하게 죽습니다.
만약 다른 주인에게 넘겨주고 싶다면, 복사가 아닌 **"소유권 이전(Move)"** 이라는 특수한 행동을 취해야만 합니다. (이건 나중에 이동 시맨틱에서 깊게 배웁니다!)

---

## 4. 하드웨어 관점: "기능이 많은데 느려지지 않을까?"

C++ 창시자의 철학(Zero-overhead Principle)이 여기서 폭발합니다.
놀랍게도 `std::unique_ptr`은 내부에 그 어떤 추가적인 쓰레기 데이터나 감시 장치를 달고 있지 않으며, 내부 코드가 전부 *inline* 처리되어 있어서 **어셈블리 단계로 번역하면 `Slime*` (생짜 포인터)를 쓰는 것과 속도/메모리가 100% 동일**합니다!

> **"안전율은 100배 올라갔는데, 성능 저하는 0% 지연인 완벽한 도구"**

이것이 모던 C++이 전 세계 모든 게임 엔진의 표준으로 다시 한번 군림하게 된 이유 중 하나입니다. 
이제 우리도 Heap을 쓸 일이 생기면 절대 `new` / `delete`를 쓰지 않고 **`std::make_unique`**를 쓸 것입니다!

## 5. 빛이 있으면 그림자도 있다: 스마트 포인터의 단점과 주의점

회창님의 날카로운 직관대로, 세상에 완벽하기만 한 도구는 없습니다. `std::unique_ptr` 등 스마트 포인터를 사용할 때 프로그래머가 겪게 될 고통(단점)은 다음과 같습니다.

### 단점 1: "복사"가 안 되어서 발생하는 극악의 학습 곡선 (Learning Curve)
`std::unique_ptr`은 태생적으로 자신이 쥐고 있는 메모리를 절대 남에게 복사(공유)해 주지 않습니다.
```cpp
void ProcessSlime(std::unique_ptr<Slime> s) { ... }

std::unique_ptr<Slime> mySlime = std::make_unique<Slime>();
ProcessSlime(mySlime); // 🚨 컴파일 에러! 매개변수로 던지는 행위 자체가 '복사'라서 막힘!
```
이걸 함수로 넘겨주거나 다른 컨테이너(`vector` 등)에 집어넣으려면 **`std::move()`라는 '이유식(이동 시맨틱)'**을 먹여서 소유권을 강제로 뺏어와야 합니다. 초보자 입장에서는 "그냥 함수에 값 하나 넘기는데 문법 에러가 왜 이렇게 많이 나지?" 하며 좌절하기 쉽습니다.

### 단점 2: 여전히 존재하는 댕글링 포인터의 위험성
`unique_ptr`이 소멸과 `delete`를 자동으로 해줄 뿐이지, **"내부의 생짜 포인터 주소를 몰래 훔쳐보는 행위"** 자체는 막지 않습니다.
```cpp
Slime* rawPtr = nullptr;
{
    std::unique_ptr<Slime> smartPtr = std::make_unique<Slime>();
    rawPtr = smartPtr.get(); // .get()으로 내부 원본 주소를 몰래 따옴
} 
// 블록 종료! smartPtr이 죽으면서 메모리를 철거함.

rawPtr->Attack(); // 🚨 크래시! 이미 폭파된 건물을 가리키는 댕글링 포인터가 됨.
```
C++은 "성능"을 위해 스마트 포인터 내부의 주소를 꺼내볼 수 있는 `.get()` 이라는 뒷구멍을 열어두었습니다. 프로그래머가 저 구멍으로 수동 조작을 하다가 실수하면 막아주지 못합니다.

### 단점 3: 또 다른 스마트 포인터, `std::shared_ptr`의 성능 오버헤드
나중에 배우게 될, 여러 명이 동시에 메모리를 공유할 수 있는 `std::shared_ptr`는 치명적인 **성능 저하(Overhead)**와 **순환 참조(A가 B를, B가 A를 가리켜서 영원히 안 죽는 버그)**라는 무서운 단점을 동반합니다. 
그래서 현업 최강 오토 엔진이라는 타이틀 이면에는, 이 스마트 포인터들을 '언제 어떤 상황에서 적절하게 골라 써야 하는가'에 대한 깊은 C++ 철학적 고민이 요구됩니다.

## 6. [심화] vector에 스마트 포인터 넣기 (`emplace_back`의 배신)

과거 모듈 04에서 복사 비용을 없애기 위해 `vector.emplace_back()`을 쓴다고 배웠습니다. 
하지만 스마트 포인터를 다룰 때는 `emplace_back`이 **예외 안전성(Exception Safety)** 측면에서 최악의 독이 됩니다.

```cpp
std::vector<std::unique_ptr<Slime>> v;

// ❌ 절대 금지 (Anti-pattern) : 끔찍한 메모리 릭 발생 가능성!
v.emplace_back(new Slime()); 
```

**왜 메모리 릭이 날까요? (실행 순서의 비극)**
1. 컴파일러는 인자인 `new Slime()`을 먼저 평가하여 힙(Heap)에 슬라임을 생성합니다.
2. 생성된 주소를 들고 `emplace_back` 내부로 들어갔는데, 아뿔싸! 벡터가 꽉 차서 이사(Reallocation)를 하려다 **메모리 부족 예외(bad_alloc)**가 터집니다.
3. 예외가 터지면서 함수가 튕겨 나가버립니다.
4. 아직 `std::unique_ptr` 캡슐 안에 들어가 보지도 못한 우리의 생짜 포인터 몬스터는, 영원히 주소를 잃어버린 채 **우주 미아(Leak)**가 됩니다.

### ✅ 모던 C++의 완벽한 해답 (`make_unique`)
그래서 모던 C++ 에서는 스마트 포인터 군단을 다룰 때 무조건 **방탄복(`make_unique`)을 먼저 입히고** 던져 넣는 방식을 씁니다.

```cpp
// 🌟 업계 표준 (안전도 100%)
v.push_back(std::make_unique<Slime>());
```
이제 실행 순서가 완벽해집니다:
1. `make_unique`가 안전하게 **임시 스마트 포인터 지역 변수**를 가장 먼저 만듭니다.
2. 0.001초 뒤에 `push_back` 과정에서 메모리 부족 수류탄이 터져서 함수가 박살 나더라도?
3. 함수 스코프를 빠져나갈 때, 1번에서 미리 만들어둔 **임시 스마트 포인터의 소멸자가 발동하여 미아가 될 뻔한 슬라임을 깔끔하게 `delete`** 해줍니다!

이것이 복사/이동을 피하겠다고 무턱대고 `emplace_back`을 쓰면 안 되는 모던 C++의 거대한 깨달음입니다.

## 7. [심화] 함수에서 포인터 반환하기 (The Factory Dilemma)

함수 내부에서 무언가(예: 슬라임)를 만들어서 बा깥으로 포인터로 전달해주고 싶을 때, 과거의 개발자들은 둘 중 하나를 반환해야 하는 지독한 딜레마에 빠졌습니다.

### 💀 1번 결말: 댕글링 포인터 (스택 주소 반환)
```cpp
Slime* CreateSlime() {
    Slime slime;       // [Stack] 함수 지역 변수로 탄생 
    return &slime;     // 🚨 [치명적 에러] 스택 메모리 주소를 넘김
}
```
함수 괄호 `}` 가 끝나는 즉시 지역 변수 `slime`은 박살 납니다. 이 주소를 사용하면 100% 확률로 크래시가 터집니다.

### 💣 2번 결말: 메모리 누수 폭탄 (힙 주소 반환)
```cpp
Slime* CreateSlime() {
    Slime* slime = new Slime(); // [Heap] 운영체제 장부에 할당
    return slime;               // ⚠️ [정상 동작, 하지만 폭탄 넘기기]
}
```
정상 작동하지만, 이걸 가져다 쓴 프로그래머가 코드 마지막에 `delete`를 치는 것을 까먹는다면 영원한 우주 미아(Memory Leak)가 됩니다.

### ✨ 모던 C++ 팩토리 패턴의 정답: `std::unique_ptr` 반환
이 공포에서 개발자들을 구원해 준 것이 바로 스마트 포인터를 반환하는 방식입니다.

```cpp
std::unique_ptr<Slime> CreateSlime() {
    // 1. 내부에서 힙 메모리에 하나 만들고, 안전 캡슐에 씌웁니다.
    std::unique_ptr<Slime> newSlime = std::make_unique<Slime>(); 
    
    // 2. 반환한다! (컴파일러가 알아서 이동(move) 처리해 줌)
    return newSlime; 
}

void main() {
    // 3. 받는 쪽에서도 똑같이 스마트 포인터로 받음!
    std::unique_ptr<Slime> boss = CreateSlime();
    // delete 신경 쓸 필요 없이, main() 함수가 닫히면 알아서 철거됨 🌟
}
```

**왜 혁명적인가요?**
함수 반환형이 `unique_ptr`이라는 것 자체가 사용자와의 **강력한 계약**입니다.
"이 슬라임 소유권 넘겼으니 맘 편히 써. 다 쓰면 알아서 지워질 거니까 `delete` 치지 마!" 

이러한 안전성 때문에, 모던 C++에서 객체를 만들어 반환하는 모든 팩토리(Factory) 함수들은 이 방식을 99% 표준으로 채택하고 있습니다.
