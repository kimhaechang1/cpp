# 개념 학습: Winsock2 환경 세팅 및 통신 흐름

네트워크 프로그래밍의 세계에 오신 것을 환영합니다. 우리가 만든 가상의 캐릭터나 아이템 정보가 다른 사람의 컴퓨터로 전달되기 위해서는 OS가 제공하는 '통신 창구'인 **소켓(Socket)**을 다룰 줄 알아야 합니다.

## 1. 샌드위치 교육법: 왜 Winsock2를 배우는가?

### 🎮 실무 프로젝트 상황
당신은 방금 멀티스레드로 동작하는 던전 레이드 시스템을 개발했습니다. 하지만 이 던전은 아직 당신의 컴퓨터 안에서만 동작합니다. 친구가 자신의 컴퓨터에서 당신의 던전에 접속해 함께 보스를 잡게 하려면 어떻게 해야 할까요?
- 내 컴퓨터의 메모리에 있는 몬스터 데이터를 친구의 컴퓨터 메모리로 복사해야 합니다.
- 전기 신호를 통해 데이터를 주고받기 위해 랜카드(NIC)를 제어해야 하는데, 이를 위해 OS가 제공하는 표준 인터페이스가 바로 **Winsock(Windows Sockets)**입니다.

---

## 2. 하드웨어와 OS의 관계

네트워크 통신은 하드웨어(랜카드), 커널(TCP/IP 스택), 그리고 유저 애플리케이션(우리 프로그램)의 유기적인 협력이 필요합니다.
- **애플리케이션**: 데이터를 보내달라고 요청함.
- **커널(OS)**: 소켓이라는 인터페이스를 통해 데이터를 패킷 단위로 쪼개고 주소를 붙임.
- **하드웨어(NIC)**: 전기 신호로 변환하여 케이블을 통해 전송함.

---

## 3. Winsock2 초기화 단계

Windows에서 네트워크 기능을 사용하려면 프로그램 시작 시 반드시 "나 네트워크 쓸 거야!"라고 신고해야 합니다.

### A. 라이브러리 링크 (Ws2_32.lib)
Winsock2 함수들은 `Ws2_32.dll`이라는 동적 라이브러리에 들어있습니다. 빌드할 때 이를 링크해주어야 합니다.
- **CMake**: `target_link_libraries(TargetName ws2_32)` 명령어를 사용합니다.

### B. WSAStartup 함수
Windows Sockets API의 사용을 시작하는 함수입니다.
- **목적**: 특정 버전의 Winsock 명세서를 사용하겠다고 알리고, OS에 관련 메모리 공간을 할당하도록 요청합니다.
- **버전**: 현재 표준인 `2.2` 버전을 주로 사용합니다. `MAKEWORD(2, 2)` 형식으로 입력합니다.

### C. WSACleanup 함수
프로그램 종료 시 빌려 썼던 네트워크 자원을 모두 반납합니다. 이를 누락하면 OS 수준에서 리소스 릭이 발생할 수 있습니다 (RAII 패턴으로 관리하는 것이 좋습니다).

---

## 4. 하드웨어 검증 및 구조체 확인

우리가 호출하는 함수가 실제로 어떤 정보를 OS로부터 받아오는지 눈으로 확인해야 합니다.

```cpp
// Winsock 초기화 시 정보를 담는 구조체
WSADATA wsaData;
int result{ WSAStartup(MAKEWORD(2, 2), &wsaData) };
```

- **wsaData.wVersion**: 실제로 할당받은 버전을 확인합니다.
- **wsaData.szDescription**: 시스템의 네트워크 벤더 정보를 확인합니다.
- **sizeof(WSADATA)**: 이 정보 묶음이 메모리 상에서 얼마나 큰 공간을 차지하는지 확인합니다.

## 5. [Q&A] 데이터의 여정: 소켓과 버퍼의 관계

훈련생님의 질문을 바탕으로 정리된 핵심 메커니즘입니다.

### Q. 내가 데이터를 보내면 상대방의 소켓과 수신 버퍼에서 어떤 일이 일어나나요?

**A.** 네, 정확합니다! 데이터는 다음과 같은 "복사의 과정"을 거칩니다.

1.  **송신자 (User -> Kernel)**: 우리 프로그램이 `send()`를 호출하면, 데이터는 **애플리케이션 메모리**에서 **OS 커널의 송신 버퍼**로 복사됩니다.
2.  **전송 (Network)**: OS가 데이터를 패킷 단위로 쪼개 네트워크망을 통해 전달합니다.
3.  **수신자 (Kernel)**: 상대방 컴퓨터의 랜카드가 전기 신호를 받으면, OS는 이를 다시 합쳐서 해당 소켓의 **커널 수신 버퍼(Receive Buffer)**에 쌓아둡니다. 
    - 이때, 상대 프로그램이 아직 데이터를 읽지 않았더라도 OS가 대신 보관해주고 있는 상태입니다. (우편함에 편지가 쌓인 것과 같음)
4.  **수신자 (Kernel -> User)**: 상대 프로그램이 `recv()`를 호출하는 순간, **OS 커널 버퍼**에 있던 데이터가 드디어 프로그램이 사용하는 **유저 메모리(버퍼)**로 복사됩니다.

### 💡 핵심 요약
- **소켓은 창구**: 프로그램과 OS 사이의 대화 창구입니다.
- **버퍼는 대기실**: 데이터가 프로그램에 전달되기 전, 속도 차이를 완충하기 위해 OS가 마련한 메모리 공간입니다.
- **핸들링의 시점**: 프로그램이 `recv()`를 통해 데이터를 자신의 메모리 영역으로 가져온 직후부터 비즈니스 로직(핸들링)이 시작됩니다.

## 6. Deep Dive: 함수와 구조체의 속사정

우리가 무심코 호출한 함수들이 OS 내부에서 어떤 일을 벌이는지 조금 더 깊게 파헤쳐 봅니다.

### 🔌 WSAStartup: 네트워크 엔진 점화
이 함수는 단순한 버전 확인용이 아닙니다.
1.  **DLL 로딩**: 실행 중인 프로그램 메모리 영역에 `ws2_32.dll`이 로드되어 있는지 확인하고, 없으면 로드합니다.
2.  **레퍼런스 카운팅**: OS는 이 DLL을 몇 개의 프로그램이 쓰고 있는지 숫자를 셉니다. `WSAStartup`이 호출될 때마다 이 숫자가 올라갑니다.
3.  **커널 자원 할당**: 프로세스별로 네트워크 상태를 관리하기 위한 내부 테이블(데이터 구조)을 힙 메모리에 생성합니다.

### 📜 WSADATA: OS가 건네주는 거래 명세서
우리가 빈 구조체를 주면 OS가 내용을 채워줍니다.
- **wVersion / wHighVersion**: 상호 합의된 버전 정보입니다.
- **iMaxSockets / iMaxUdpDg**: (과거의 유물) 한 번에 열 수 있는 최대 소켓 수와 UDP 패킷 크기입니다. 현재는 OS가 동적으로 관리하므로 하위 호환성을 위해 존재합니다.
- **szDescription**: "Microsoft Windows Sockets 2.0" 같은 시스템의 이름표입니다.

### 🧹 WSACleanup: 뒷정리 및 종료
- **카운트 감소**: DLL 사용 숫자를 하나 줄입니다.
- **메모리 해제**: 숫자가 0이 되면 메모리에서 `ws2_32.dll`을 완전히 내리고, 관련 커널 자원을 모두 삭제합니다.
- **중요성**: 이걸 호출하지 않으면 소켓 통신을 종료한 후에도 OS가 "아직 이 프로그램이 네트워크를 쓰고 있네?"라고 착각하여 메모리를 계속 붙잡고 있게 됩니다 (**Resource Leak**).

## 7. ⚠️ 훈련생을 위한 핵심 Q&A

### Q. WSAStartup을 하면 바로 서버가 켜지는 건가요?
**A. 아닙니다!** `WSAStartup`은 단지 "Windows 네트워크 엔진에 시동을 거는 행위"일 뿐입니다. 소켓을 만들거나 포트를 배정받는 등의 구체적인 서버 활동은 이후 `socket()`, `bind()` 등의 함수를 별도로 호출해야 시작됩니다.

### Q. 이 단계에서 소켓 ID나 포트 번호가 배정되나요?
**A. 아직은 아무것도 없습니다.**
- **소켓 디스크립터(ID)**: 내 손에 전화기가 아직 없는 상태입니다. (`socket()` 호출 시 생성)
- **포트 번호(Port)**: 내 전화번호를 등록하지 않은 상태입니다. (`bind()` 호출 시 생성)

지금은 오직 **"네트워크 기능을 사용할 자격"**만 얻은 상태라고 이해하시면 완벽합니다.
