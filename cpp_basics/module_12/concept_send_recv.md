# 데이터 송수신의 핵심: send & recv

## 🏢 실무 상황 (The Scenario)
서버와 클라이언트가 서로 연결되었습니다(`connect`/`accept` 성공). 이제 클라이언트가 서버에게 "안녕! 나는 전사 캐릭터야"라고 말을 걸고, 서버는 그걸 받아서 "반가워! 게임 월드에 접속한 걸 환영해"라고 대답해주어야 합니다.

---

## 📤 1. 데이터 보내기: `send()`

소켓이라는 빨대에 데이터를 밀어 넣는 과정입니다.

```cpp
const char* msg = "Hello Server!";
int sendLen = send(targetSocket, msg, (int)strlen(msg), 0);
```

- **`targetSocket`**: 누구에게 보낼 것인가? (서버는 `clientSocket`, 클라이언트는 `myLocalSocket`)
- **반환값 (`int`)**: 실제로 몇 바이트가 빨대에 성공적으로 들어갔는지를 알려줍니다.

---

## 📥 2. 데이터 받기: `recv()`

빨대 끝에서 데이터가 나오기를 기다렸다가 컵(Buffer)에 담는 과정입니다.

```cpp
char buffer[1024];
int recvLen = recv(targetSocket, buffer, sizeof(buffer), 0);
```

### ⚠️ 중요: `recv`의 3가지 반환값
- **`recvLen > 0`**: 이만큼의 바이트를 성공적으로 받았습니다! (잔치 시작)
- **`recvLen == 0`**: 상대방이 연결을 끊었습니다. (종료 처리 필요)
- **`recvLen < 0`**: 통신 에러가 발생했습니다. (`WSAGetLastError()` 확인!)

---

## 🌊 3. TCP의 "스트림" 성격 (Packet Fragmentation)

이것이 오늘 수업의 **가장 중요한 포인트**입니다.

TCP는 데이터에 **경계가 없습니다.**
- 보내는 쪽에서 "Hello"라고 한 번에 `send` 했어도,
- 받는 쪽에서는 네트워크 상황에 따라 "He"를 먼저 받고, 조금 있다가 "llo"를 받을 수도 있습니다.

> [!WARNING]
> 즉, `recv`를 한 번 호출했다고 해서 상대방이 보낸 전체 데이터가 다 왔다고 확신하면 안 됩니다! 우리가 원하는 만큼 데이터가 다 올 때까지 반복해서 받아야 할 수도 있습니다.

---

## 🔍 4. 하드웨어 시각화 (Buffer Flow)

데이터가 이동할 때의 모습입니다.

```text
[Client]                      [OS Kernel Buffer]                    [Server]
  send("ABC")  ------>  [ A | B | C | _ | _ ]  ------>  recv(buffer, 1024)
  (데이터 밀어내기)        (잠시 보관하는 창고)           (창고에서 꺼내오기)
```

만약 서버가 너무 바빠서 `recv`를 제때 호출하지 못하면, OS 커널 내부의 수신 버퍼(창고)가 꽉 차게 되고, 그때부터는 클라이언트의 `send`가 멈추게 됩니다. (흐름 제어)

---

## ❓ Q&A (자가 진단)

1.  **`recv`를 호출했는데 데이터가 아직 도착 안 했으면 어떻게 되나요?**
    - 정답: 기본 설정에서는 데이터가 올 때까지 프로그램이 그 줄에서 멈춰서 기다립니다. 이를 **블로킹(Blocking)** 모드라고 합니다.
3.  **문자열 IP를 다시 원래대로 돌릴 수 있나요?**
    - 정답: **`inet_ntop`** (Network to Presentation) 함수를 쓰면 됩니다. 숫자 덩어리인 IP를 우리가 읽기 편한 "127.0.0.1"로 바꿔줍니다.
4.  **`INET_ADDRSTRLEN`은 왜 쓰나요?**
    - 정답: IPv4 주소를 문자열로 담을 때 필요한 최소 배열 크기(16)를 나타내는 안전장치입니다.
