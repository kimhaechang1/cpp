# [부록] 엔디언(Endianness)과 변환 함수의 이해

네트워크 프로그래밍에서 데이터의 "바이트 순서"를 맞추는 것은 서로 다른 언어를 쓰는 두 사람이 대화하기 위해 통역사를 두는 것과 같습니다.

---

## 1. 엔디언 종류: Little vs Big

숫자 `0x12345678` (4바이트)이 메모리에 저장되는 실제 모습입니다.

| 방식 | 설명 | 메모리 저장 순서 (낮은 주소 → 높은 주소) | 비유 |
| :--- | :--- | :--- | :--- |
| **Little Endian** | 낮은 자릿수(꼬리)부터 저장 | `0x78` \| `0x56` \| `0x34` \| `0x12` | 거꾸로 읽기 (Intel CPU) |
| **Big Endian** | 높은 자릿수(머리)부터 저장 | `0x12` \| `0x34` \| `0x56` \| `0x78` | 똑바로 읽기 (Network 표준) |

---

## 2. 0x1E61 (7777)의 네트워크 여행기

우리가 포트 번호 **7777**을 보낼 때, 바이트들이 실제로 어떤 순서로 이동하는지 보여주는 "바이트 지도"입니다.

```text
[단계 1] 송신측 (Intel CPU / Little-Endian)
-----------------------------------------------------------
  * CPU 레지스터  :       0x1E61 (7777)
  * 메모리 적재   : [ 0x61 ] [ 0x1E ]  <-- 꼬리 바이트가 앞 주소에!
        (변환 전)    (addr)  (addr+1)

  * htons() 변환  : [ 0x1E ] [ 0x61 ]  <-- 전송을 위해 강제로 뒤집음
        (변환 후)    (addr)  (addr+1)
-----------------------------------------------------------
            |
            | (전선/네트워크망을 타고 흐름)
            v
[단계 2] 네트워크 상의 데이터 (Big-Endian 표준)
-----------------------------------------------------------
  * 전송 순서     :  [ 0x1E ] ----> [ 0x61 ]
                   (먼저 도착)      (나중에 도착)
-----------------------------------------------------------
            |
            | (상대방 컴퓨터에 도착)
            v
[단계 3] 수신측 (Intel CPU / Little-Endian)
-----------------------------------------------------------
  * 메모리 기록   : [ 0x1E ] [ 0x61 ]  <-- 받은 순서대로 일단 기록
        (변환 전)    (addr)  (addr+1)

  * ntohs() 변환  : [ 0x61 ] [ 0x1E ]  <-- 내 CPU가 읽기 좋게 뒤집음
        (변환 후)    (addr)  (addr+1)

  * CPU가 읽기    :       0x1E61 (7777)  <-- 정확한 값 복구 성공!
-----------------------------------------------------------
```

### 🚢 여행 단계별 포인트
1.  **송신측 메모리**: 인텔 CPU는 꼬리(`0x61`)를 앞에 둡니다. 그대로 보내면 네트워크는 `0x611E`라고 오해하게 됩니다.
2.  **`htons()`의 역할**: 전송 직전, 바이트 순서를 수동으로 뒤집어서 **`1E-61`** 순서로 만듭니다 (값 자체는 일시적으로 깨져 보이지만 전송을 위해 참습니다!).
3.  **네트워크 전송**: 망 위에서는 가장 먼저 들어온 `1E`가 상위 바이트가 되어 **`0x1E61`**이라는 온전한 값이 유지됩니다.
4.  **수신측 `ntohs()`**: 받은 데이터를 내 컴퓨터 방식(꼬리부터)으로 다시 정렬하여, 내 CPU가 읽었을 때 다시 **7777**이 되도록 만듭니다.

---

## 3. 왜 변환 함수가 필요한가?

인터넷(Network)은 **Big-Endian**을 표준으로 채택했습니다. 하지만 우리가 쓰는 대부분의 개인 PC(Intel, AMD)는 **Little-Endian**을 사용합니다. 

만약 변환 없이 데이터를 보내면, 받는 쪽에서는 숫자가 완전히 뒤집힌 채로 해석하게 되어 **엉뚱한 IP 주소나 포트 번호**로 통신하게 됩니다.

---

## 3. 변환 함수 4형제 (N-H-L-S)

함수 이름만 보면 용도를 바로 알 수 있도록 명명 규칙(Naming Convention)이 짜여 있습니다.

### 💡 이름의 의미
- **h**: **H**ost (내 컴퓨터)
- **n**: **N**etwork (인터넷 망)
- **to**: **to** (~로)
- **s**: **S**hort (16비트 / 2바이트, 주로 **Port 번호**용)
- **l**: **L**ong (32비트 / 4바이트, 주로 **IP 주소**용)

### 🛠️ 함수 일람표
| 함수 | 방향 | 주 대상 | 역할 |
| :--- | :--- | :--- | :--- |
| **`htons()`** | Host → Network (Short) | 포트 번호 | 보낼 때 2바이트 뒤집기 |
| **`htonl()`** | Host → Network (Long) | IP 주소 | 보낼 때 4바이트 뒤집기 |
| **`ntohs()`** | Network → Host (Short) | 포트 번호 | 받았을 때 2바이트 다시 뒤집기 |
| **`ntohl()`** | Network → Host (Long) | IP 주소 | 받았을 때 4바이트 다시 뒤집기 |

---

## 4. 이식성(Portability)의 마법

"내 컴퓨터가 이미 Big-Endian(네트워크랑 동일) 방식이라면 어떡하죠?"

그 경우 위 함수들은 **아무런 작업도 하지 않고(Identity)** 원래 값을 그대로 반환합니다. 덕분에 우리는 상대방의 CPU 아키텍처가 무엇인지 고민할 필요 없이, **위 함수들을 습관적으로 사용하기만 하면 전 세계 어디서나 돌아가는 코드**를 완성할 수 있습니다.
