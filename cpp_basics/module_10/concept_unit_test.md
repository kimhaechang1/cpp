# [MODULE 10] 단위 테스트 (Unit Test) & Google Test

자동화된 도구(GTest)를 배우기 전, 우리는 **"왜 테스트가 필요한지"**와 **"도구 없이 테스트하면 어떤 고통이 따르는지"**를 먼저 이해해야 합니다.

---

## 1. 단위 테스트(Unit Test)란 무엇인가? 🎯

단위 테스트는 시스템의 **가장 작은 조각(단위, 주로 함수나 클래스)**이 의도한 대로 동작하는지 고립된 환경에서 검증하는 작업입니다.

*   **목적**: "내가 고친 코드가 기존 기능을 망가뜨리지 않았는가?"를 1초 만에 확인하기 위함입니다.
*   **핵심**: 외부의 DB나 파일 시스템 등에 의존하지 않고, 오직 해당 함수의 로직 그 자체(Logic Integrity)만 검증합니다.

---

## 2. 도구 없이 테스트하기: <cassert> 활용 🧱

라이브러리 없이 C++ 표준만으로도 테스트는 가능합니다. 바로 `<cassert>` 헤더의 **`assert()`** 함수를 사용하는 것이죠.

```cpp
#include <iostream>
#include <cassert> // (1) 표준 어설션 헤더 포함

int Add(int a, int b) { return a + b; }

int main() {
    // (2) 수동 테스트 작성
    assert(Add(10, 20) == 30); // 맞으면 통과, 틀리면 프로그램 강제 종료!
    assert(Add(-1, 1) == 0);
    
    std::cout << "모든 테스트 성공!" << std::endl;
    return 0;
}
```

### 🚨 "원시적 테스트"의 치명적인 한계
하지만 이 방식은 **현업**에서는 절대 쓰지 않습니다. 왜일까요?
1.  **Stop-at-First-Fail**: 첫 번째 테스트가 틀리면 프로그램이 터져서(Aborted), 뒤에 있는 나머지 99개의 테스트 결과를 확인할 수 없습니다.
2.  **가독성 부족**: 상세한 리포트가 전혀 없습니다. (그저 core dump만 남길 뿐...)
3.  **관리의 어려움**: 테스트가 1,000개라면 `main` 함수가 수만 줄이 되어 버립니다.

## <a id="qa"></a> 3. 🧐 Q&A (중요!)

### Q1. "Stop-at-First-Fail"은 테스트들이 병렬적으로 실행될 수 없어서인가요?
**반은 맞고 반은 틀립니다.**
*   **원시 방식**: 한 `main` 함수 안에서 `assert()`를 쓰면 단일 스레드로 동작하며, 하나만 죽어도 프로세스 자체가 종료되므로 뒤를 확인할 수 없습니다.
*   **프레임워크 방식**: 테스트를 독립된 객체로 관리하여 하나가 실패하더라도 '보고'만 하고 다음으로 넘깁니다. 원한다면 각 독립된 테스트들을 병렬(Parallel)로 실행할 수도 있습니다.

### Q2. 코어 덤프(Core Dump)가 무엇인가요? 📂
Crashing 시점의 메모리 및 CPU 상태를 파일로 기록한 블랙박스 스냅샷입니다.

### Q3. 코어 덤프는 자동으로 생성되나요? 🤖
OS 설정에 따라 다릅니다! Linux는 `ulimit` 설정으로 자동 생성되게 할 수 있고, Windows는 디버거가 가로채서 보여주는 방식을 선호합니다.

### Q4. 특정 테스트를 별도의 함수로 분리해서 main에서 호출해도 되지 않나요?
**완벽합니다! 그것이 바로 테스트 프레임워크가 작동하는 기본 원리입니다.** 
하지만 수동으로 하면 두 가지 불편함이 남습니다:
1.  **등록(Registration) 누락**: 새 테스트 함수를 만들 때마다 `main`에 가서 호출 코드를 한 줄씩 추가해 줘야 합니다. 하나라도 깜빡하면 그 테스트는 영원히 안 돌아갑니다.
2.  **여전한 강제 종료**: 함수로 분리했더라도, `main`에서 순서대로 호출하는 구조라면 앞선 함수에서 `assert()`가 터질 때 프로그램 전체가 죽는 건 똑같습니다. 즉, 전체 보고서를 볼 수 없습니다.

---

## 4. Google Test (GTest)의 솔루션 🧪

GTest는 위 Q4에서 언급한 **"자동 등록"**과 **"실패 시에도 계속 진행"**을 매크로와 정적 등록(Static Registration) 기법으로 우아하게 해결했습니다.
