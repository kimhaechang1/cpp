# Module 10: 프로페셔널 도구 - 협업 기초 (Git)

## 🎮 실무 프로젝트 상황 (샌드위치 교육법)
혼자 게임을 만들 때는 상관없지만, 넥슨이나 엔씨소프트 같은 큰 회사에서는 수십 명의 개발자가 하나의 소스 코드를 공유합니다.

*   **문제**: 누군가 코드를 수정하다가 실수로 게임이 안 켜지게 만들었다면?
*   **문제**: 두 명이 동시에 똑같은 파일을 수정했다면?
*   **해결**: Git이라는 "시간 여행 장치"와 "평행 세계 관리자"를 사용하여 코드의 역사를 기록하고 협업합니다.

---

## 📦 1. `.gitignore` (불필요한 데이터 거르기)

C++ 프로젝트를 빌드하면 수많은 중간 파일(`.obj`, `.exe`, `.pdb`)이 생깁니다. 이 파일들은 내 컴퓨터의 환경에 종속적이므로 서버에 올릴 필요가 없습니다.

**필수 제외 항목:**
1.  **빌드 결과물**: `*.exe`, `*.out`, `*.bin`
2.  **중간 컴파일 파일**: `*.obj`, `*.o`, `*.pdb`
3.  **IDE 설정**: `.vs/`, `.vscode/`, `out/`, `build/`
4.  **외부 라이브러리 (바이너리)**: `vcpkg_installed/` (vcpkg.json으로 관리하므로)

## 🌿 2. Git 버전 관리 전략 (Workflow)

실무에서 가장 많이 쓰이는 **Git Flow**의 핵심만 뽑아봅시다.

### 1) 브랜치(Branch) 나누기
*   **main**: 제품 출시용. 가장 안정적인 상태.
*   **develop**: 다음 버전을 위한 통합소. 개발자들이 각자 만든 기능을 여기서 합침.
*   **feature/xxx**: 특정 기능 개발용 '개인 연습장'. (예: `feature/quest-system`)

### 2) 커밋(Commit) - "체크포인트 저장"
의미 있는 단위로 쪼개서 저장해야 나중에 문제가 생겼을 때 특정 지점으로 되돌아가기 쉽습니다.
*   ❌ "오늘 작업 끝" (나쁜 예)
*   ✅ "Quest: 몬스터 처치 퀘스트 스켈레톤 구현" (좋은 예)

### 3) 코드 리뷰 & 머지(Merge)
개인 브랜치에서 만든 코드를 `develop`에 합치기 전, 팀원이 코드를 검사하는 과정을 **Pull Request (PR)**라고 합니다. 여기서 버그를 잡고 "승인"을 받아야 메인 코드에 합쳐질 수 있습니다.

---

## 🏗️ 하드웨어/시스템 무결성 가이드
Git 저장소 크기를 줄이는 것은 클라우드 비용 절감뿐만 아니라, 팀원들이 프로젝트를 내려받는(Clone) 속도에도 직결됩니다. 
불필요한 `build` 폴더를 실수로 올렸다면 `git rm --cached` 명령어로 원격 저장소에서만 제거할 수 있습니다.
