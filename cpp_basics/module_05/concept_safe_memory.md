# Module 05 보충: 메모리 안전성 (Memory Safety)

## 1. 댕글링 포인터 (Dangling Pointer) 👻
"매달려 있는 포인터"라는 뜻입니다. 이미 `delete` 되어서 사라진 메모리를 여전히 가리키고 있는 포인터를 말합니다.

### 1.1 위험성
```cpp
int* p = new int(10);
delete p; // 메모리 반납 완료!

// ... 100줄 뒤 ...

*p = 20; // 으악! 이미 반납한 땅에 침범 (Use-After-Free 버그)
// 운 좋으면 크래시, 운 나쁘면 잘 돌아가다가 나중에 데이터 오염 발생
```

### 1.2 `delete p`의 진실 (중요 ⭐)
**Q: `delete p`를 하면 `p` 변수도 사라지나요?**
A: **아니요!** `p`가 가리키던 **"힙 메모리의 집(데이터)"만 철거**됩니다.
`p`가 들고 있던 **"주소값(쪽지)"**은 그대로 남아있습니다. (철거된 집 주소를 들고 있는 상태)
이 상태의 포인터를 **댕글링 포인터**라고 부릅니다.

### 1.3 `delete pp` (이중 포인터)를 하면요? 😱
**Q: `SafeDelete(int** pp)` 함수 안에서 `delete pp`를 하면 `pp->p->data` 중 `p`가 사라지나요?**
A: **아니요! 프로그램이 터집니다!** (Crash)
- `pp`는 `main` 함수 스택에 있는 `p`의 주소를 가지고 있습니다. (스택 메모리 주소)
- `delete`는 오직 **"힙(Heap) 메모리 주소"**만 받을 수 있습니다.
- 스택 주소를 `delete` 하려고 했으니, 운영체제가 "내 땅 아닌데 왜 건드려?" 하고 프로그램을 강제 종료시킵니다.
- **정답**: `delete *pp`를 해야 합니다. (`*pp`는 힙에 있는 데이터를 가리키는 주소니까요!)

### 1.4 `pp = nullptr`은 안 하나요?
**Q: `*pp`는 지우고 나서 `nullptr`로 만드는데, 왜 `pp`는 `nullptr`로 안 만드나요?**
A: `pp`는 **SafeDelete 함수가 끝나면 어차피 사라질 지역 변수(Stack)**이기 때문입니다.
- **중요한 건 바깥 세상(`main`)에 있는 변수**를 `nullptr`로 만드는 겁니다. (`*pp = nullptr`의 역할)
- `pp` 자체는 `SafeDelete` 함수가 끝나면 스택에서 자동 소멸되므로, 굳이 `nullptr`을 넣을 필요가 없습니다. (넣어도 의미 없음)

그래서 반드시 `p = nullptr`로 "쪽지의 내용도 지워줘야" 안전합니다.

## 2. 안전한 해제 전략: `nullptr` 세탁 🧼
메모리를 해제하자마자 포인터에 `nullptr`(주소 0번지)을 대입하여, **"더 이상 유효하지 않음"**을 명시합니다.

```cpp
delete p;
p = nullptr; // 이제 p는 아무데도 가리키지 않음 (SAFE)

// ...
if (p != nullptr) { // 안전하게 체크 가능
    *p = 20;
}
```

> **습관**: `delete`와 `nullptr` 대입은 **한 세트**라고 생각하세요.

## 3. Use-After-Return (스택의 댕글링)
동적 할당뿐만 아니라 스택에서도 댕글링 포인터가 발생할 수 있습니다.

```cpp
int* DangerousFunc() {
    int local = 100;
    return &local; // [치명적 에러] 지역 변수는 함수 끝나면 죽는데, 그 주소를 리턴?
}
```
함수가 끝나면 `local` 변수는 사라집니다. 리턴받은 포인터는 **사라진 유령**을 가리키게 됩니다. 절대 금지! ❌

> **💡 미래의 고민거리 (팩토리 패턴의 딜레마)**
> "그럼 함수 바깥으로 객체를 전달하고 싶을 땐 스택 대신 힙(`new`) 주소를 반환하면 되는 거 아닌가요?"
> 맞습니다. 하지만 인간이 매번 `delete`를 쳐줘야 한다는 끔찍한 협업 문제(메모리 누수 폭탄)가 발생합니다.
