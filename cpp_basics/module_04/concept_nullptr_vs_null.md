# Module 4 보충: NULL과 nullptr의 치명적 차이

## 📌 겉보기엔 같아 보이는 "비어있는 주소"
과거 C 언어 시절부터 오랫동안 사용해 온 **`NULL`**과 모던 C++11에 등장한 **`nullptr`**는 둘 다 포인터가 비어있음을 나타낼 때 사용합니다.
하지만 내부적으로 이 둘은 **"단순 숫자"**와 **"절대적 포인터"**라는 하늘과 땅 차이를 가지고 있습니다.

## 📌 1. `NULL`의 정체 (재앙의 시작)
C언어 프로그래머들이 즐겨 쓰던 `NULL`의 내부 구현을 까보면 그저 정수 `0`으로 치환되는 매크로에 불과합니다.
```cpp
#define NULL 0
```
컴파일러의 눈에 `NULL`은 메모리 주소가 아니라 그저 **숫자 `0`**입니다.

### 🚨 `NULL`이 일으키는 끔찍한 오버로딩 에러
숫자 0이기 때문에, C++의 핵심 기능인 **함수 오버로딩(Overloading)**을 만났을 때 대참사가 벌어집니다.

```cpp
void DeleteMonster(int monsterID) {
    cout << "1. 정수 아이디로 몬스터 삭제!";
}

void DeleteMonster(Monster* ptr) {
    cout << "2. 포인터 주소로 몬스터 삭제!";
}
```

이제 "포인터를 비워서(NULL) 몬스터 삭제를 확인해야지!" 하고 아래처럼 호출하면 어떻게 될까요?
```cpp
DeleteMonster(NULL); 
```
**[결과]** `"1. 정수 아이디로 몬스터 삭제!"` 가 실행되어 버립니다! 😱
개발자는 분명히 2번 포인터 버전이 호출될 줄 알았지만, 컴파일러는 `NULL`을 **정수 `0`**으로 인식해버리기 때문에 엉뚱하게 `int` 버전 함수를 호출해 버리는 최악의 논리적 버그(Silent Bug)가 터집니다.

## 📌 2. 구원자 `nullptr` 의 등장
C++ 위원회는 이 사태를 막기 위해 C++11에서 **오직 메모리 주소 용도로만 취급되는 전용 키워드**이자 새로운 데이터 타입(`std::nullptr_t`)인 `nullptr`를 도입했습니다.

```cpp
DeleteMonster(nullptr);
```
이렇게 호출하면, 컴파일러는 "이건 무조건 포인터구나!"라고 100% 확신하므로 완벽하게 2번 `DeleteMonster(Monster* ptr)` 버전을 호출하게 됩니다.

> 💡 **Best Practice 핵심 룰**
> "모던 C++ 코드를 작성할 때, 대문자 `NULL`은 뇌리에서 완전히 지워버리고 무조건 소문자 `nullptr`만 사용하세요. C++ 커뮤니티의 철칙입니다."
