# Module 4/5 보충: std::span과 복사(Copy)의 비밀

## 📌 std::span은 복사되지 않나요?

만약 함수가 배열의 일부 구간(View)을 계산해서 던져줄 때, 반환 타입으로 참조(`&`) 없이 일반 `std::span<int>`를 **값으로 반환(Return by value)** 한다고 가정해 봅시다.

```cpp
std::span<int> GetFirstHalf(std::vector<int>& data) {
    std::span<int> s{data};
    return s.subspan(0, data.size() / 2);
}
```

참조나 포인터가 아님에도 불구하고 데이터 자체는 복사되지 않습니다. 그 이유는 무엇일까요?

### 1. `std::span`의 내부는 포인터에 불과하다
`std::span`의 내부 구조를 열어보면 아주 단순한 C 언어 스타일의 구조체와 같습니다.
```cpp
template <typename T>
class span {
    T* ptr;     // 데이터가 시작하는 메모리 주소 (8바이트)
    size_t len; // 볼 수 있는 원소의 개수 (8바이트)
};
```
즉, `span` 객체 자체의 크기는 **단 16바이트**(64비트 시스템 기준) 뿐입니다.

### 2. 가벼운 껍데기와 무거운 알맹이
`span`을 함수의 반환 반환값으로 넘기거나(반환타입), 매개변수로 전달할 때(값에 의한 복사 호출) 일어나는 것은 오직 **"명함(16바이트)의 복사"**일 뿐입니다.
수백 메가바이트의 우물(원본 배열)이 있더라도, 알맹이인 원본 데이터 배열 자체는 **결코 복사되지 않습니다.** 단지 우물을 가리키는 이정표만 전달되는 것이죠.

### 3. 결론: Best Practice
따라서 C++ Core Guidelines에서는 `std::span`을 함수의 인자나 반환타입으로 사용할 때 참조(`const std::span&`)를 쓰지 말고, **값으로 전달(`std::span`)**하는 것을 강력하게 권장합니다. (참조를 쓰면 포인터의 포인터를 만드는 셈이 되어 캐시 효율성(Cache Locality) 측면에서 오히려 CPU 성능에 악영향을 줍니다.)
